# 24 | JTBD框架 Jobs to Be Done

## TL;DR (3分钟速读)

**Jobs to Be Done（JTBD，待完成任务理论）** 是一种以用户任务为中心的产品思维框架，核心观点是：**用户不是购买产品，而是"雇佣"产品来完成某个任务**。

**核心要点：**
- **Job（任务）**：用户想要达成的目标或解决的问题（不是产品功能）
- **AI产品应用**：理解用户"雇佣"AI助手的真实原因（如豆包不是"聊天工具"，而是"快速完成文案创作的助手"）
- **竞争重新定义**：JTBD视角下，真正的竞争对手可能来自不同行业（如Kimi的竞争对手可能是"人工助理"，而不只是其他AI）

**实战建议：**
1. **访谈用户**：不问"你觉得我们产品怎么样"，而问"上次使用我们产品，你当时想解决什么问题？"
2. **识别Job**：用"当___时候，我想___，以便___"的句式描述用户任务
3. **竞争分析**：找出所有能完成同一Job的解决方案（包括非AI产品）

## 1. JTBD理论基础

### 1.1 什么是Jobs to Be Done？

**Jobs to Be Done（JTBD）** 是由哈佛商学院教授Clayton Christensen提出的创新理论，核心观点是：

> **"顾客不是购买产品，而是'雇佣'（hire）产品来完成某个任务（job）。"**

**经典案例：快餐店的奶昔销售**
- **传统思维**：顾客买奶昔是因为口味好、价格便宜
- **JTBD思维**：研究发现，早上通勤时段买奶昔的顾客，真实Job是"让漫长无聊的通勤时间过得快一点，同时不会把车弄脏"（奶昔可以单手握住，慢慢喝，不像面包会掉渣）

**洞察**：这些顾客的真正"竞争对手"不是其他奶昔品牌，而是**香蕉、百吉饼、咖啡**等所有能完成"通勤打发时间"这个Job的产品。

### 1.2 JTBD为什么重要？

**传统产品思维的局限**：
- **以产品为中心**："我们的AI助手有多模态能力、支持长文本..."（聚焦功能）
- **以用户属性为中心**："25-35岁白领喜欢我们的产品"（聚焦人口统计）
- **问题**：无法解释为什么同一类用户有时用、有时不用产品

**JTBD思维的优势**：
- **以任务为中心**："当用户需要快速写一份会议纪要时，他们会'雇佣'我们的AI助手"（聚焦场景）
- **洞察**：真正的竞争不是"豆包 vs Kimi"，而是"AI助手 vs 人工记录 vs 飞书妙记"

**对AI产品的价值**：
1. **重新定义竞争**：发现隐藏的竞争对手（如人工、传统软件、甚至"什么都不做"）
2. **指导功能优先级**：优先开发最能完成Job的功能
3. **精准定位**：不是"我们是AI助手"，而是"我们帮你10秒完成周报"

## 2. JTBD核心概念

### 2.1 Job（任务）的定义

**Job** = 用户在特定情境下想要达成的目标或解决的问题

**Job的三要素**：

```
┌──────────────────────────────────────┐
│  当 [情境]                            │
│  我想 [目标]                          │
│  以便 [期望结果]                      │
└──────────────────────────────────────┘
```

**AI产品示例**：
- **豆包**："当我写抖音短视频脚本时（情境），我想快速生成3个备选方案（目标），以便选出最吸引人的开头（期望结果）"
- **Kimi**："当我读一份80页的行业报告时（情境），我想快速提取核心结论（目标），以便向老板汇报重点（期望结果）"

**Job的特征**：
- **稳定性**：Job本身不会随技术变化（人类一直需要"快速获取信息"）
- **情境依赖**：同一个人在不同情境下有不同Job（通勤时听播客 vs 工作时查资料）
- **与解决方案无关**：Job不应包含具体产品名称（"我想快速写文案"，而不是"我想用ChatGPT写文案"）

### 2.2 Job的分类

#### **Functional Job（功能性任务）**
**定义**：用户想要完成的具体功能性目标

**AI产品示例**：
- **文案创作**："写一篇小红书种草笔记"
- **信息检索**："找出这份合同的风险条款"
- **代码生成**："用Python写一个爬虫脚本"

#### **Emotional Job（情感性任务）**
**定义**：用户想要获得的情感体验或感受

**AI产品示例**：
- **减少焦虑**："快速得到答案，不用担心搜索半天找不到"
- **提升自信**："生成的文案让我看起来很专业"
- **获得认可**："用AI生成的PPT赢得老板表扬"

#### **Social Job（社交性任务）**
**定义**：用户想要塑造的社会形象或人设

**AI产品示例**：
- **展现效率**："在团队里成为'工作效率高'的人"
- **展现专业**："在客户面前体现'数据分析专家'的形象"
- **展现创新**："在公司内成为'AI应用先锋'"

**三种Job的关系**：
```
┌────────────────────────────────────────┐
│  Functional Job（核心）                 │
│  "完成具体任务"                         │
│                                        │
│  ┌──────────────────────────────────┐ │
│  │ Emotional Job（辅助）             │ │
│  │ "获得良好感受"                    │ │
│  │                                  │ │
│  │ ┌────────────────────────────┐   │ │
│  │ │ Social Job（加分项）        │   │ │
│  │ │ "塑造社会形象"              │   │ │
│  │ └────────────────────────────┘   │ │
│  └──────────────────────────────────┘ │
└────────────────────────────────────────┘
```

**实战应用**：
- **产品核心价值**：必须满足Functional Job（如AI必须真的能生成可用文案）
- **差异化竞争**：通过Emotional Job突出（如"用豆包写文案，不焦虑"）
- **品牌建设**：通过Social Job强化（如"用Notion AI的人都是高效能人士"）

### 2.3 Job的粒度

**Job粒度** = Job描述的抽象程度

```
┌─────────────────────────────────────────┐
│ 粒度太粗（过于抽象）                     │
│ "我想提高工作效率"                      │
│ → 太宽泛，无法指导产品设计               │
└─────────────────────────────────────────┘
           │
           ▼
┌─────────────────────────────────────────┐
│ 合适粒度（可执行）                       │
│ "我想10分钟内完成周报"                  │
│ → 可以设计具体功能                       │
└─────────────────────────────────────────┘
           │
           ▼
┌─────────────────────────────────────────┐
│ 粒度太细（过于具体）                     │
│ "我想点击'生成周报'按钮"                │
│ → 这是功能描述，不是Job                  │
└─────────────────────────────────────────┘
```

**判断标准**：合适粒度的Job应该**与解决方案无关，但可以衡量完成效果**。

## 3. JTBD vs 传统需求分析

### 3.1 对比：JTBD vs 用户故事 vs 用户画像

| **维度** | **JTBD** | **用户故事** | **用户画像** |
|---------|---------|------------|------------|
| **关注焦点** | 用户想完成的任务 | 用户想要的功能 | 用户的属性特征 |
| **描述方式** | "当___时，我想___，以便___" | "作为___，我想___，以便___" | "25-35岁白领，年薪30万" |
| **稳定性** | 高（任务不变） | 中（功能随技术变化） | 低（属性无法预测行为） |
| **竞争分析** | 找出所有能完成Job的方案 | 聚焦同类产品功能对比 | 聚焦同类用户群体 |
| **适用阶段** | 产品定位、功能优先级 | 功能设计、开发排期 | 市场营销、用户研究 |

**示例对比**：

| **方法** | **描述** |
|---------|---------|
| **JTBD** | "当我参加线上会议时，我想自动生成会议纪要，以便会后快速回顾要点" |
| **用户故事** | "作为产品经理，我想要AI自动记录会议，以便不用手动记笔记" |
| **用户画像** | "张小明，28岁，互联网公司产品经理，每天参加3-5个会议" |

**三者结合使用**：
1. **JTBD**：确定产品定位（解决哪个Job？）
2. **用户故事**：设计具体功能（如何解决Job？）
3. **用户画像**：制定营销策略（向谁推广？）

### 3.2 JTBD的独特价值

**传统需求分析的盲点**：

**场景1：功能列表陷阱**
- **传统思维**："竞品有15个功能，我们要做20个才能赢"
- **JTBD思维**："用户真正需要完成的Job只有3个，聚焦把这3个做到极致"

**场景2：用户属性迷思**
- **传统思维**："我们的目标用户是25-35岁白领"
- **问题**：同样是25-35岁白领，有人用豆包写文案，有人用Kimi读论文，有人根本不用AI
- **JTBD思维**："我们的目标用户是'需要快速生成创意文案的内容创作者'（Job定义，不是年龄定义）"

**场景3：竞品分析局限**
- **传统思维**："我们的竞品是ChatGPT、Claude、Gemini"（只看同类AI产品）
- **JTBD思维**："用户完成'写周报'这个Job的竞争方案包括：AI助手、Word模板、复制上周周报、请同事代写、甚至不写"（跨品类竞争）

## 4. JTBD在AI产品中的应用

### 4.1 用JTBD重新定义AI产品

**传统AI产品定位**（功能导向）：
- "我们是一款AI对话助手，支持多轮对话、多模态输入、代码生成..."

**JTBD视角的AI产品定位**（任务导向）：
- "当你需要快速完成周报时，我们帮你10分钟搞定"（豆包）
- "当你需要深度理解长文档时，我们帮你一次处理200页"（Kimi）
- "当你需要准确的搜索结果时，我们给你可靠来源"（Perplexity）

**差异**：
- 功能导向：用户需要先理解产品能力，再想如何使用
- 任务导向：用户直接知道"什么时候用这个产品"

### 4.2 AI产品的常见Job类型

#### **内容创作Job**
| **Job描述** | **AI产品** | **传统解决方案** |
|-----------|----------|---------------|
| 快速生成抖音短视频脚本 | 豆包、通义千问 | 自己写、参考爆款视频、雇佣编剧 |
| 润色邮件让语气更专业 | ChatGPT、文心一言 | 自己改、用Grammarly、请同事帮忙 |
| 生成PPT大纲 | Gamma、Beautiful.ai | 参考模板、复制旧PPT、用模板库 |

#### **信息获取Job**
| **Job描述** | **AI产品** | **传统解决方案** |
|-----------|----------|---------------|
| 快速找到问题答案 | Perplexity、豆包 | 百度搜索、问同事、查文档 |
| 深度理解长文档 | Kimi、Claude | 自己读、找人总结、用PDF阅读器 |
| 学习新领域知识 | ChatGPT、文心一言 | 看书、上课、看B站教程 |

#### **效率提升Job**
| **Job描述** | **AI产品** | **传统解决方案** |
|-----------|----------|---------------|
| 自动记录会议纪要 | 飞书妙记、腾讯会议AI | 手动记录、录音后整理、不记录 |
| 快速生成代码 | Cursor、GitHub Copilot | 自己写、复制Stack Overflow、找同事 |
| 批量处理Excel数据 | ChatGPT Code Interpreter | 手动处理、写Python脚本、找数据分析师 |

### 4.3 用JTBD指导功能优先级

**案例：某AI写作助手功能排期**

**候选功能**：
1. 长文本生成（支持10000字）
2. 多风格切换（正式/轻松/幽默）
3. 多模态输入（上传图片生成文案）
4. 实时协作（多人同时编辑）
5. SEO优化建议

**JTBD分析**：

| **功能** | **对应Job** | **Job频率** | **优先级** |
|---------|-----------|-----------|-----------|
| 多风格切换 | "让文案适配不同场景（商务邮件/社交媒体）" | ⭐⭐⭐⭐⭐（每天） | P0（必做） |
| 长文本生成 | "一次性完成长篇内容（如白皮书）" | ⭐⭐（每月） | P2（次要） |
| SEO优化 | "让文章获得更多搜索流量" | ⭐⭐⭐⭐（每周） | P1（重要） |
| 多模态输入 | "根据产品图片生成电商文案" | ⭐⭐⭐（每周） | P1（重要） |
| 实时协作 | "团队共同创作文案" | ⭐（偶尔） | P3（暂缓） |

**决策**：优先开发"多风格切换"（高频Job）和"SEO优化"（刚需Job），暂缓"实时协作"（低频Job）。

**传统功能优先级方法问题**：
- 可能因为"长文本生成"技术难度高、差异化明显而优先开发
- 但用户真实高频Job是"快速切换文案风格"

## 5. JTBD用户访谈方法

### 5.1 传统访谈 vs JTBD访谈

| **问题类型** | **传统访谈** | **JTBD访谈** |
|------------|------------|------------|
| **产品评价** | "你觉得我们产品怎么样？" | "上次用我们产品，你当时想解决什么问题？" |
| **功能需求** | "你希望产品有哪些功能？" | "上次完成这个任务时，哪个环节最困难？" |
| **竞品对比** | "你觉得我们比竞品好在哪？" | "为什么选择我们而不是XX？当时在什么情况下？" |
| **痛点挖掘** | "产品有哪些不满意的地方？" | "上次没用我们产品，而是用了XX，为什么？" |

**JTBD访谈的核心原则**：
- **聚焦具体事件**：不问假设，只问"上一次实际发生的事"
- **挖掘情境**：详细了解当时的场景、心理、环境
- **探索替代方案**：了解用户尝试过的所有解决方案

### 5.2 JTBD访谈框架（Switch Interview）

**Switch Interview** = 了解用户从旧方案"切换"到新方案的过程

**访谈流程**（6步法）：

```
┌─────────────────────────────────────────┐
│ 1. First Thought（首次意识）             │
│ "你第一次意识到需要解决这个问题是什么时候？" │
└──────────────────┬──────────────────────┘
                   │
┌──────────────────▼──────────────────────┐
│ 2. Passive Looking（被动搜索）            │
│ "之后你做了什么？有关注过解决方案吗？"    │
└──────────────────┬──────────────────────┘
                   │
┌──────────────────▼──────────────────────┐
│ 3. Active Looking（主动搜索）             │
│ "什么事件让你决定'我必须解决这个问题'？" │
└──────────────────┬──────────────────────┘
                   │
┌──────────────────▼──────────────────────┐
│ 4. Deciding（决策）                       │
│ "你评估了哪些方案？为什么选择我们？"      │
└──────────────────┬──────────────────────┘
                   │
┌──────────────────▼──────────────────────┐
│ 5. First Use（首次使用）                  │
│ "第一次用的时候感觉怎样？符合预期吗？"    │
└──────────────────┬──────────────────────┘
                   │
┌──────────────────▼──────────────────────┐
│ 6. Ongoing Use（持续使用）                │
│ "现在还在用吗？什么情况下会用？"          │
└─────────────────────────────────────────┘
```

### 5.3 JTBD访谈示例（AI写作助手）

**访谈对象**：某市场运营，使用豆包生成社交媒体文案

**访谈记录**：

**1. First Thought（首次意识）**
- **Q**: "你第一次意识到'需要AI帮助写文案'是什么时候？"
- **A**: "大概3个月前，老板要求每天发3条小红书，我一个人根本写不过来，经常加班到晚上10点。"

**2. Passive Looking（被动搜索）**
- **Q**: "当时你有没有想过找工具帮忙？"
- **A**: "有啊，看到同事在用ChatGPT，但我试了下全是英文，不太会用。后来在抖音刷到豆包的广告，但没马上试。"

**3. Active Looking（主动搜索）**
- **Q**: "什么事情让你决定'我必须找个AI工具'？"
- **A**: "有一次连续加班一周，周五晚上11点还在写文案，突然崩溃了。第二天马上下载了豆包、Kimi、文心一言，一个个试。"

**4. Deciding（决策）**
- **Q**: "你试了这么多，为什么最后选了豆包？"
- **A**: "Kimi适合长文档，我不需要；文心一言生成的文案太正式，不适合小红书；豆包生成的文案口语化，而且可以直接说'写一条小红书种草笔记'，它就懂我要什么。"

**5. First Use（首次使用）**
- **Q**: "第一次用豆包的时候感觉怎样？"
- **A**: "有点惊喜，10秒钟就生成了3个版本，我改改标题就能发。但也有问题，有时候生成的内容太浮夸，需要手动改。"

**6. Ongoing Use（持续使用）**
- **Q**: "现在还在用吗？什么情况下会用？"
- **A**: "每天都用，主要是早上批量生成3-5条备选文案，下午根据热点手动改。如果是重要的商业合作文案，我还是自己写，因为AI生成的不够有个性。"

**JTBD提取**：
```
当 [每天需要发3条小红书，时间紧迫]
我想 [快速生成3-5条备选文案]
以便 [不加班也能完成KPI，保留精力做重要文案]
```

**洞察**：
- **Job粒度**："快速生成备选文案"（不是"写完美文案"）
- **竞争对手**："自己加班写""复制爆款文案""雇实习生"
- **关键场景**："批量生成日常文案"（不是"重要商业文案"）
- **情感Job**："减少加班焦虑""保留创意精力"

## 6. JTBD竞争分析

### 6.1 重新定义竞争

**传统竞争分析**：
- 只看同类产品（豆包 vs Kimi vs ChatGPT）
- 对比功能参数（上下文长度、响应速度、价格）

**JTBD竞争分析**：
- 看所有能完成同一Job的方案（包括非AI、人工、甚至"不做"）
- 对比完成Job的效果（速度、成本、质量、体验）

**豆包"写小红书文案"的竞争对手**：

| **竞争方案** | **类型** | **优势** | **劣势** |
|------------|---------|---------|---------|
| 豆包AI | AI工具 | 10秒生成、成本低 | 需要人工润色、缺乏个性 |
| 自己写 | 人工 | 最有个性、完全可控 | 耗时长（30分钟/条）、创意枯竭 |
| 复制爆款 | 人工 | 效率中等、质量有保证 | 缺乏原创性、可能侵权 |
| 雇实习生 | 外包 | 节省时间、可批量生产 | 成本高（5000元/月）、管理成本 |
| Kimi长文本 | AI工具 | 适合长文章 | 不适合短文案、响应慢 |
| 不发内容 | 不做 | 成本为0 | 失去流量、老板不满意 |

**洞察**：
- **真正的威胁**：不是Kimi（定位不同），而是"复制爆款"（效率高、质量可控）
- **破局点**：让豆包生成的文案"有爆款套路"，减少"复制爆款"的需求

**JTBD竞争地图** = 在"完成Job的效果"维度上对比所有方案

**战略洞察**：

### 6.3 JTBD竞争优势分析

**评估AI产品在Job上的竞争力**：

| **评估维度** | **问题** | **豆包示例** |
|------------|---------|------------|
| **功能匹配度** | 产品能否完成这个Job？ | ✅ 可以生成小红书文案 |
| **完成速度** | 比其他方案快多少？ | ✅ 10秒 vs 自己写30分钟 |
| **完成质量** | 效果是否达到用户预期？ | ⚠️ 需要润色，但可接受 |
| **易用性** | 用户学习成本多高？ | ✅ 输入一句话即可 |
| **成本** | 相比其他方案便宜吗？ | ✅ 免费 vs 雇人5000元/月 |
| **情感体验** | 使用过程是否愉悦？ | ✅ 减少加班焦虑 |
| **切换成本** | 从旧方案切换难度？ | ✅ 下载即用，无切换成本 |

**竞争优势判断**：
- **绝对优势**：速度（10秒 vs 30分钟）、成本（免费 vs 雇人）
- **相对优势**：易用性（输入一句话 vs 学习ChatGPT）
- **劣势**：质量（需要润色 vs 自己写完全可控）

**战略决策**：
- **强化绝对优势**：保持"10秒生成"的速度承诺
- **弥补劣势**：增加"一键润色"功能，提升质量

**豆包的主要应用场景**（基于行业观察）：

豆包DAU已突破1亿([36氪](https://www.36kr.com/p/3528783696338049))，主要应用场景包括：

## 8. JTBD与产品创新

### 8.1 用JTBD发现创新机会

**创新机会来源**：
1. **未被满足的Job**：用户有Job，但所有现有方案都不理想
2. **被过度满足的Job**：现有方案功能过多，用户只需要简化版
3. **被低估的Job**：市场忽视了某个重要Job

**AI产品创新示例**：

#### **机会1：未被满足的Job**
- **Job**："快速验证AI生成内容的准确性"
- **现状**：用户用ChatGPT生成内容后，需要手动验证事实
- **创新机会**：推出"自动事实核查AI"（如Perplexity的引用来源功能）
- **趋势方向**：AI产品增加内容溯源功能，标注关键信息的来源

#### **机会2：被过度满足的Job**
- **Job**："快速得到简单问题的答案"
- **现状**：ChatGPT功能强大，但对于简单问答（如"北京明天天气"）过于复杂
- **创新机会**：推出"极简AI搜索"（输入问题，直接显示答案，无需对话）
- **趋势方向**：搜索引擎探索"AI快答"模式，简化用户获取信息的路径

#### **机会3：被低估的Job**
- **Job**："让AI生成的内容有我的个人风格"
- **现状**：所有AI生成的文案千篇一律，缺乏个性
- **创新机会**：推出"AI个性化写作"（学习用户历史文案，生成符合个人风格的内容）
- **趋势方向**：AI从"一刀切"向"千人千面"演进，实现基于用户行为和环境数据的个性化内容生成([澎湃新闻](https://m.thepaper.cn/newsDetail_forward_32471796))

### 8.2 JTBD创新画布

**创新画布** = 系统性发现创新机会的工具

```
┌──────────────────────────────────────────┐
│ Job: 快速完成周报                         │
├──────────────────────────────────────────┤
```

**创新方向示例**："AI周报自动化"

- **差异化**：不是通用AI助手，而是专注"周报"这一个垂直Job
- **实现路径**：基于企业协作平台的工作流自动化和AI智能体能力

## 9. JTBD常见误区

### 9.1 误区1：把功能当作Job

**错误示例**：
- **错误Job**："我想用AI对话"
- **问题**："AI对话"是功能，不是用户想完成的任务

**正确示例**：
- **正确Job**："我想快速得到某个问题的答案"（对话只是手段）

**判断技巧**：如果Job描述中包含产品名称或功能名称，通常是错误的。

### 9.2 误区2：Job粒度过粗

**错误示例**：
- **错误Job**："我想提高工作效率"
- **问题**：太宽泛，无法指导产品设计

**正确示例**：
- **正确Job**："我想10分钟内完成周报，以便有更多时间做深度工作"

**判断技巧**：好的Job应该可以**衡量完成效果**（如"10分钟内完成"）。

### 9.3 误区3：忽略情感和社交Job

**错误示例**：
- 只关注功能性Job（如"生成文案"），忽略情感Job（如"减少焦虑"）

**正确示例**：
- 豆包的完整Job："快速生成文案（功能）+ 减少加班焦虑（情感）+ 展现效率（社交）"

**实战应用**：
- **营销文案**：强调情感Job（"告别加班，用豆包10秒生成文案"）
- **产品设计**：满足社交Job（生成文案后显示"你比90%用户更高效"）

### 9.4 误区4：JTBD访谈问假设问题

**错误示例**：
- "如果我们推出XX功能，你会用吗？"（假设问题，用户往往回答"会"，但实际不用）

**正确示例**：
- "上次你用我们产品，当时想解决什么问题？"（具体事件，用户真实行为）

**JTBD访谈原则**：
- 只问**过去实际发生的事**，不问假设
- 问**"为什么"**，而不是**"你觉得怎么样"**

### 9.5 误区5：以为JTBD只适用于新产品

**错误认知**：
- "我们产品已经成熟了，不需要JTBD分析"

**正确认知**：
- JTBD适用于产品全生命周期：
  - **新产品**：定义产品定位（解决哪个Job？）
  - **成熟产品**：优化功能优先级（哪些Job被低估？）
  - **衰退产品**：发现转型机会（用户转向哪些新Job？）

**潜在应用**：传统OA软件可通过JTBD分析，识别用户Job是否从"流程审批"转向"快速决策"，进而推出AI辅助决策功能实现转型。

## 💡 实战洞察

### 洞察1：AI产品的Job往往不是"对话"

很多AI产品自我定位为"对话助手"，但用户真实的Job通常是：
- **不是**："我想和AI对话"
- **而是**："我想快速得到答案/完成文案/理解文档"

**实战建议**：
- 不要在营销文案中强调"对话能力"（用户不关心）
- 强调"完成具体任务的能力"（如"10秒生成文案""一次处理200页文档"）

### 洞察2：用户"放弃使用"也是一种竞争方案

JTBD分析中，最容易被忽视的竞争对手是**"用户选择不做这个Job"**：
- **豆包案例**：有些用户放弃写小红书，改做抖音短视频（不需要文案）
- **Kimi案例**：有些用户放弃深度阅读长文档，直接问同事结论

**实战建议**：
- 在用户访谈中，问"有没有干脆放弃这个任务"
- 如果大量用户放弃，说明这个Job本身可能不值得做

### 洞察3：中国AI用户的Job具有文化特殊性

相比国际市场，中国用户的Job有独特特征：
- **高频碎片化Job**：通勤、等电梯、午休时使用（适合短内容生成）
- **强社交属性Job**："让朋友圈/小红书内容显得有品位"（社交Job权重高）
- **工具+生态Job**："与飞书/钉钉/微信无缝集成"（孤立工具难以成功）

**实战建议**：
- 针对中国市场，强化碎片化场景支持（如"一句话生成文案"）
- 在产品设计中，满足社交Job（如生成内容自带精美排版）
- 深度集成中国主流工具（飞书/钉钉/微信/抖音）

## 📚 核心术语表

| **英文术语** | **中文翻译** | **定义** |
|------------|------------|---------|
| **Jobs to Be Done (JTBD)** | 待完成任务理论 | 用户不是购买产品，而是"雇佣"产品来完成某个任务 |
| **Job** | 任务 | 用户在特定情境下想要达成的目标或解决的问题 |
| **Hire** | 雇佣 | 用户选择使用某个产品/服务来完成Job |
| **Fire** | 解雇 | 用户停止使用某个产品/服务，转向其他方案 |
| **Functional Job** | 功能性任务 | 用户想要完成的具体功能性目标 |
| **Emotional Job** | 情感性任务 | 用户想要获得的情感体验或感受 |
| **Social Job** | 社交性任务 | 用户想要塑造的社会形象或人设 |
| **Switch Interview** | 切换访谈 | 了解用户从旧方案"切换"到新方案的访谈方法 |
| **Job Granularity** | 任务粒度 | Job描述的抽象程度（过粗/合适/过细） |
| **Competition Map** | 竞争地图 | 在"完成Job的效果"维度上对比所有方案的可视化图表 |

   - D. "我想10分钟内生成3个抖音脚本备选"

   - 提示：哪个是核心？哪个是加分项？

### 案例分析题

4. **某用户说"我想用Kimi读论文"，这是一个好的Job描述吗？如果不是，如何改进？**
   - 提示：Job不应包含产品名称

## 🎯 实战练习

### 练习1：识别你的产品核心Job（30分钟）

**步骤**：
1. 选择一款AI产品（豆包/Kimi/你负责的产品）
2. 访谈3-5个真实用户（或分析用户评价）
3. 提取核心Job（用"当___时，我想___，以便___"格式）
4. 区分Functional Job、Emotional Job、Social Job

**输出**：一份Job清单（3-5个核心Job）

### 练习2：JTBD竞争分析（45分钟）

**任务**：
1. 选择一个具体Job（如"快速完成周报"）
2. 列出所有能完成这个Job的方案（至少5个，包括非AI、人工、不做）
3. 对比各方案在速度/质量/成本/易用性的表现
4. 绘制JTBD竞争地图

**输出**：一份竞争分析报告（包含竞争地图）

### 练习3：JTBD用户访谈实战（持续）

**任务**：
1. 找1个真实用户进行JTBD访谈（30分钟）
2. 使用5.2节的访谈框架（6步法）
3. 录音并转录（飞书妙记/讯飞听见）
4. 提取用户的核心Job和洞察

**输出**：一份完整的访谈记录+Job提取分析
