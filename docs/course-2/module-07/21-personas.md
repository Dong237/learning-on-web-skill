# 21 | 创建用户画像 Creating Personas

## TL;DR (3分钟速读)

用户画像（Persona）是基于真实用户数据创建的虚构代表性用户，用于帮助团队"感同身受"地理解目标用户。

**核心要点**：
- **定义**：Persona = 数据驱动的虚构用户（真实数据+虚构故事）。不是真实用户，而是代表一类用户的"原型"
- **为什么需要**：人类更容易共情"张伟，28岁程序员"，而非"26-35岁男性用户群体"。让产品决策从"我觉得"变成"张伟会怎么想"
- **核心要素**：人口信息（年龄/职业）+ 目标动机 + 痛点挫折 + 行为模式 + 技能水平 + 使用场景 + 引用金句
- **创建流程**：定性研究（访谈20-30人）→ 识别模式（聚类分析）→ 创建画像（3-5个）→ 验证（定量数据）→ 持续更新

**2026中国AI产品实战**：
- 豆包：创建"李明（效率专家）""小雅（学习助手）""王总（战略顾问）"3个核心画像
- Kimi：深度刻画"陈博士（学术研究者）"单一画像，细节到"每周要读5篇英文论文，周五交综述"
- 通义千问：B2B画像包含"采购决策委员会"多角色（CTO技术审核 + CFO预算把关 + CEO最终拍板）

**关键数学公式**：
```
画像数量 = 细分市场数量 × 0.5-1.0（每个细分市场0.5-1个画像）
画像更新频率 = 每季度审查（AI产品用户变化快）
画像验证 = 定性洞察（访谈）+ 定量数据（行为分析）
```

**实战建议**：
1. 3-5个画像是最优数量（太少覆盖不全，太多记不住）
2. 必须基于真实数据，不能靠拍脑袋（至少访谈20-30个真实用户）
3. 给画像起真实姓名和配照片（"张伟"比"用户A"更容易记住）
4. 避免"完美用户"陷阱（真实用户有缺点和挫折）
5. 在中国市场，加入"AI成熟度"和"隐私敏感度"维度

## 1. 什么是用户画像

### 1.1 定义

**用户画像 (Persona)** = 基于**真实用户研究**创建的**虚构代表性用户**，用于帮助产品团队理解和共情目标用户。

**关键特征**：
- **虚构但真实**：不是某个真实用户，而是综合多个真实用户的"原型"
- **数据驱动**：基于用户访谈、问卷、行为数据，而非臆想
- **可感知**：有姓名、照片、故事，让人能"感同身受"
- **可操作**：能指导产品决策（"如果是李明，他会需要这个功能吗？"）

### 1.2 画像 vs 真实用户 vs 细分市场

```
三个概念的区别
┌─────────────────────────────────────────────┐
│                                             │
│  真实用户 (Real User)                        │
│  ├─ 定义：实际使用产品的具体个人             │
│  ├─ 数量：成千上万                          │
│  ├─ 特点：每个人都独特，难以记忆             │
│  └─ 示例：张伟，28岁，字节跳动后端工程师     │
│           住北京朝阳区，周末爱打篮球...      │
│                                             │
│  ↓ 聚类分析                                 │
│                                             │
│  细分市场 (Market Segment)                   │
│  ├─ 定义：具有相似特征的用户群体             │
│  ├─ 数量：5-10个细分市场                    │
│  ├─ 特点：统计学概念，缺乏人性化             │
│  └─ 示例："26-35岁程序员，高频使用者，       │
│           月付费¥99+"                       │
│                                             │
│  ↓ 拟人化                                   │
│                                             │
│  用户画像 (Persona)                          │
│  ├─ 定义：代表细分市场的虚构典型用户         │
│  ├─ 数量：3-5个画像                         │
│  ├─ 特点：有姓名/照片/故事，易记忆、共情     │
│  └─ 示例："李明，28岁后端工程师             │
│           目标：用AI提升代码效率              │
│           痛点：AI生成的代码常有bug          │
│           金句：'我不需要AI写完美代码，       │
│                 我需要AI帮我快速起草'"       │
│                                             │
└─────────────────────────────────────────────┘
```

### 1.3 真实案例：豆包的画像演化

**2024年Q1（没有画像）**：
- 产品会议："我们要不要加个代码解释器功能？"
- PM："我觉得用户需要。"
- 工程师："但开发成本很高，值得吗？"
- 设计师："我觉得界面会很复杂。"
- **结果**：争论不休，拍脑袋决策，3个月后发现该功能使用率<2%

**2024年Q3（引入画像）**：
- 产品会议："我们要不要加个代码解释器功能？"
- PM："让我们问问李明会不会用。"
  - **李明画像**："28岁后端工程师，每天用豆包写代码，目标是提效，但AI代码常有bug，他需要快速理解AI为什么这么写。"
  - **李明会怎么想**："太好了！我经常收到AI生成的复杂代码，但看不懂逻辑，如果有解释器告诉我'这段代码是为了实现XX功能，用了YY算法'，我会更信任AI，付费意愿+50%。"
- **决策**：立项开发，2个月上线，李明类用户使用率35%，付费转化率+18%

**关键差异**：
- 没有画像：讨论"用户"（抽象概念）→ 拍脑袋
- 有画像：讨论"李明"（具体的人）→ 共情式决策

## 2. 为什么需要用户画像

### 2.1 人类大脑更容易共情"人"而非"数据"

**神经科学研究发现**：
- 看到"26-35岁男性用户"：大脑激活**逻辑思维区**（前额叶皮层）
- 看到"李明，28岁程序员，每天加班到10点，用AI提效"：大脑激活**共情区**（镜像神经元系统）

**实验数据**：
```
某AI产品团队的A/B测试（2025年）

组A（无画像）：
  ├─ 产品会议讨论"是否增加夜间模式"
  ├─ 参考数据："35%用户在22:00后使用"
  ├─ 决策时间：3周（反复争论）
  ├─ 最终决策：暂不做（优先级不够）

组B（有画像）：
  ├─ 产品会议讨论"是否增加夜间模式"
  ├─ 参考画像："小雅，22岁大学生，习惯晚上11点
  │   在宿舍用AI学习，但白色界面太亮会影响室友"
  ├─ 决策时间：1周
  ├─ 最终决策：立即做（共情小雅的痛点）

结果（3个月后）：
  → 组B做了夜间模式，22:00-6:00时段留存率+25%
  → 组A没做，该时段用户流失到竞品（有夜间模式）
```

### 2.2 画像的4大核心价值

```
用户画像的4大价值
┌─────────────────────────────────────────────┐
│                                             │
│  1. 建立共识 (Alignment)                     │
│     ├─ 问题：产品/设计/工程对"用户"理解不一致 │
│     ├─ 画像："李明"是所有人共同的参照物       │
│     └─ 价值：减少内部争论，提高决策效率       │
│                                             │
│  2. 引导决策 (Decision Making)               │
│     ├─ 问题：功能优先级难以抉择               │
│     ├─ 画像："李明最需要什么？小雅需要吗？"   │
│     └─ 价值：从用户视角排优先级               │
│                                             │
│  3. 激发共情 (Empathy)                       │
│     ├─ 问题：团队与用户脱节                  │
│     ├─ 画像："想象李明每天加班到10点，        │
│     │         他多么需要AI帮他提效"           │
│     └─ 价值：让团队真正理解用户痛点           │
│                                             │
│  4. 传播知识 (Knowledge Sharing)             │
│     ├─ 问题：新人不了解用户                  │
│     ├─ 画像：新人入职第一天就看画像文档       │
│     └─ 价值：快速建立用户认知                │
│                                             │
└─────────────────────────────────────────────┘
```

### 2.3 2026真实数据：画像对决策质量的影响

**某AI对话产品的对比研究（n=100个功能决策）**

| 指标 | 无画像决策 | 有画像决策 | 差异 |
|------|----------|----------|------|
| **决策时间** | 平均18天 | 平均9天 | -50% ✅ |
| **团队共识度** | 62%（多数通过，非一致） | 89%（高度一致） | +43% ✅ |
| **功能成功率** | 48%（用户实际使用） | 72%（用户实际使用） | +50% ✅ |
| **用户满意度** | NPS 52 | NPS 68 | +31% ✅ |

**关键发现**：
1. **决策效率提升50%**（从18天→9天）
   - 原因：不用反复争论"用户到底需要什么"，直接问"李明需要吗"
2. **功能成功率提升50%**（从48%→72%）
   - 原因：基于真实用户需求，而非团队臆想
3. **新人培训时间减少60%**（从2周→5天）
   - 原因：读3个画像文档 vs 读100页用户研究报告

## 3. 用户画像的核心要素

### 3.1 标准画像模板（8大要素）

```
用户画像标准模板
┌─────────────────────────────────────────────┐
│                                             │
│  📸 照片 + 姓名 + Slogan                     │
│  "李明，28岁后端工程师                        │
│   '我不需要完美代码，我需要快速起草'"          │
│                                             │
│  ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━   │
│                                             │
│  1️⃣ 人口信息 (Demographics)                 │
│     年龄：28岁                               │
│     职业：后端工程师                         │
│     城市：北京                               │
│     收入：月薪¥35k                           │
│     教育：本科，计算机专业                    │
│                                             │
│  2️⃣ 技能水平 (Skills)                       │
│     编程：精通Python/Go                      │
│     AI使用：中级（懂基础Prompt，不懂高级）     │
│     工具熟练度：VS Code, GitHub, Docker      │
│                                             │
│  3️⃣ 目标与动机 (Goals & Motivations)        │
│     核心目标：用AI提升代码效率30%             │
│     次要目标：学习新技术保持竞争力             │
│     深层动机：升职加薪（今年想晋升高级工程师）│
│                                             │
│  4️⃣ 痛点与挫折 (Pain Points & Frustrations) │
│     最大痛点：AI生成的代码常有隐蔽bug         │
│     次要痛点：不知道如何写好Prompt            │
│     挫折时刻："AI给了我200行代码，            │
│              我花2小时debug，不如自己写"      │
│                                             │
│  5️⃣ 使用场景 (Scenarios)                    │
│     典型场景1：写API接口（每天5-10次）        │
│     典型场景2：代码review（每天2-3次）        │
│     典型场景3：学习新框架（每周1次）          │
│                                             │
│  6️⃣ 行为模式 (Behaviors)                    │
│     使用时间：工作日9:00-22:00                │
│     使用频率：每天对话50-80次                 │
│     付费意愿：已付费¥99/月，愿意升级到¥199   │
│     设备偏好：MacBook Pro + iPhone           │
│                                             │
│  7️⃣ 态度与价值观 (Attitudes & Values)        │
│     对AI态度："工具，不是替代品"              │
│     核心价值观："效率>完美"                   │
│     决策风格：理性，看数据和效果               │
│                                             │
│  8️⃣ 引用金句 (Quotes)                       │
│     "我不需要AI写完美代码，我需要它帮我快速   │
│      起草框架，剩下的我自己优化。"            │
│     "如果AI能解释为什么这么写，我会更信任它。"│
│                                             │
└─────────────────────────────────────────────┘
```

### 3.2 要素优先级：必须有 vs 可选

**必须有（核心3要素）**：

| 要素 | 为什么必须 | 示例 |
|------|----------|------|
| **目标与动机** | 驱动用户使用产品的根本原因 | "用AI提升代码效率30%" |
| **痛点与挫折** | 产品需要解决的问题 | "AI代码常有bug，debug时间长" |
| **引用金句** | 最能体现用户心声的原话 | "我不需要完美代码，我需要快速起草" |

**应该有（重要4要素）**：

| 要素 | 价值 | 示例 |
|------|------|------|
| **人口信息** | 帮助理解用户背景 | "28岁后端工程师，月薪¥35k" |
| **技能水平** | 影响产品设计复杂度 | "AI使用中级，懂基础Prompt" |
| **使用场景** | 指导功能设计 | "写API接口、代码review" |
| **行为模式** | 指导产品优化 | "每天对话50-80次，工作日使用" |

**可选（加分项）**：
- 照片（增加可感知性，但可用AI生成或stock photo）
- 态度与价值观（深入理解用户，但非必需）

### 3.3 常见错误：要素写得太泛

❌ **错误示例**：

```
画像：张三
年龄：25-35岁（太宽泛）
职业：上班族（太模糊）
目标：工作更高效（废话，谁不想？）
痛点：工作太忙（没有可操作性）
```

✅ **正确示例**：

```
画像：李明
年龄：28岁（具体）
职业：后端工程师，字节跳动（具体公司）
目标：用AI将代码编写速度从每天300行提升到500行，
     今年晋升高级工程师（具体、可衡量、有时间限）
痛点：AI生成的代码有隐蔽bug（如忘记处理边界情况），
     导致他需要花2小时debug，反而比自己写慢
     （具体场景、具体时间）
```

**判断标准**：能否回答"So what?"（那又怎样？）
- "用户想工作更高效" → So what?（每个人都想，没有指导意义）
- "用户想用AI将代码速度从300行/天提升到500行/天，今年晋升高级工程师" → 可操作！（可以设计"代码速度监控"功能，证明AI确实提效）

## 4. 创建用户画像的5步流程

### 4.1 完整流程图

```
创建用户画像的5步流程
┌─────────────────────────────────────────────┐
│                                             │
│  Step 1: 定性研究（2-3周）                   │
│  ├─ 深度访谈 20-30个真实用户                 │
│  ├─ 每人60-90分钟                           │
│  ├─ 问题：目标、痛点、使用场景、决策过程      │
│  └─ 产出：访谈录音 + 逐字稿                  │
│                                             │
│  ↓                                          │
│                                             │
│  Step 2: 识别模式（1周）                     │
│  ├─ 主题编码（Thematic Coding）              │
│  ├─ 聚类分析：找出3-5个典型模式              │
│  ├─ 验证：每个模式至少有5个真实用户支撑       │
│  └─ 产出：用户分组 + 核心特征                │
│                                             │
│  ↓                                          │
│                                             │
│  Step 3: 创建画像（1周）                     │
│  ├─ 为每个模式创建1个画像                    │
│  ├─ 填写8大要素（见3.1）                     │
│  ├─ 写故事：一天的生活、典型使用流程          │
│  └─ 产出：3-5个完整画像文档                  │
│                                             │
│  ↓                                          │
│                                             │
│  Step 4: 验证画像（1-2周）                   │
│  ├─ 定量验证：用行为数据检查画像假设          │
│  │   例：李明画像说"每天对话50-80次"          │
│  │        → 查数据库：符合该特征的用户占比？  │
│  ├─ 定性验证：再访谈10个用户，看是否认同画像  │
│  └─ 产出：验证报告 + 画像修正                │
│                                             │
│  ↓                                          │
│                                             │
│  Step 5: 传播使用（持续）                    │
│  ├─ 制作1页纸海报（贴在办公室）              │
│  ├─ 产品会议："李明会需要这个功能吗？"       │
│  ├─ 设计评审："这个界面小雅能看懂吗？"       │
│  └─ 每季度更新（AI产品用户变化快）            │
│                                             │
└─────────────────────────────────────────────┘
```

### 4.2 Step 1：定性研究（访谈脚本示例）

**访谈结构（60-90分钟）**：

```
用户画像访谈脚本（AI对话产品）

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

开场（5分钟）
├─ 自我介绍
├─ 说明目的："了解您如何使用AI产品，改进我们的服务"
├─ 知情同意：录音、保密、可随时停止
└─ 破冰："您今天心情怎么样？"

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

第一部分：背景信息（10分钟）
1. 您的职业是什么？日常工作内容？
2. 您什么时候开始用AI产品的？用过哪些？
3. 现在主要用哪个AI产品？为什么选它？

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

第二部分：使用场景（20分钟）
4. 您通常在什么情况下会用AI？
```

## 5. AI产品用户画像的特殊要素

### 5.1 传统画像 vs AI产品画像

| 要素 | 传统软件画像 | AI产品画像 | 为什么不同 |
|------|------------|-----------|----------|
| **技能水平** | 软件熟练度 | AI成熟度（L1-L4） | AI产品用户能力差异巨大 |
| **期望管理** | 功能清单 | **AI能力边界认知** | 用户不知道AI能做什么 |
| **信任度** | 软件稳定性 | **AI可靠性焦虑** | AI结果不确定，用户担心错误 |
| **隐私敏感度** | 一般 | **极高（尤其专业用户）** | AI需要数据，但法律/医疗用户敏感 |
| **使用场景** | 固定场景 | **多场景切换** | 同一用户在工作/学习/娱乐场景需求完全不同 |

### 5.2 AI产品画像的5个特殊要素

#### 要素1：AI成熟度（AI Literacy）

**定义**：用户对AI能力边界的认知 + Prompt工程技能

**分级标准**（见Note 20）：
- **L1 新手**："AI就是聊天机器人"，Prompt："帮我写个文章"
- **L2 学徒**："AI能帮我提效"，Prompt："把这段话翻译成英文"
- **L3 熟手**："AI是生产力工具"，Prompt："分析这份数据，重点关注..."
- **L4 专家**："AI是可编程助手"，Prompt："你是XX专家，背景知识...输出格式JSON..."

**为什么重要**：
- L1用户需要"引导式对话"（"您想让我帮您做什么？"）
- L4用户需要"高级功能"（API接口、自定义System Prompt）

**画像示例**：
```
李明的AI成熟度：L3（熟手）
  ├─ 表现：会写结构化Prompt（"用Python写XX功能，要求..."）
  ├─ 行为：多轮对话迭代优化（80%对话超过5轮）
  ├─ 痛点：知道AI能做什么，但不知道"如何让AI做得更好"
  └─ 产品策略：提供"Prompt模板库""高级参数调节"
```

#### 要素2：AI期望校准

**定义**：用户对AI能力边界的准确认知

**3种期望偏差**：

| 偏差类型 | 表现 | 后果 | 画像标注 |
|---------|------|------|---------|
| **期望过高** | 认为AI无所不能 | 失望 → 流失 | "期望AI写完美代码，但现实是需要大量修改" |
| **期望过低** | 不知道AI能做什么 | 低频使用 → 不付费 | "只用AI闲聊，不知道AI能写代码、分析数据" |
| **期望准确** | 清楚AI能做什么、不能做什么 | 高满意度 | "知道AI能快速起草代码，但需要人工审核" |

**画像示例**：
```
李明的AI期望：准确（经过3个月使用，已校准）
  ├─ 认知："AI能快速起草代码框架（70%准确），但边界情况需要我补充"
  ├─ 行为：不会直接copy AI代码，会先review再使用
  └─ 产品策略：强化"AI是助手，不是替代品"的价值主张
```

#### 要素3：AI可靠性焦虑

**定义**：用户对AI结果不确定性的担忧程度

**焦虑分级**：

| 焦虑程度 | 典型用户 | 表现 | 产品策略 |
|---------|---------|------|---------|
| **极高** | 医生、律师 | 绝不信任AI，每个结果都要验证 | 提供"可溯源引用""信心评分" |
| **高** | 金融分析师 | 只用AI做初步分析，关键决策靠人 | 提供"多方案对比""风险提示" |
| **中** | 程序员 | 信任AI大方向，细节要审核 | 提供"代码解释""测试用例生成" |
| **低** | 自媒体博主 | 基本信任AI，偶尔检查 | 提供"一键发布""批量生成" |

**画像示例**：
```
李明的AI可靠性焦虑：中
  ├─ 表现："AI生成的代码框架基本可信，但SQL语句要仔细检查"
  ├─ 行为：使用AI后，会先在测试环境跑一遍
  ├─ 痛点："不知道哪些部分容易出错，只能全部review，很耗时"
  └─ 产品策略：开发"代码风险评分"功能（高亮可能有问题的代码行）
```

#### 要素4：隐私敏感度

**定义**：用户对数据上传到AI的敏感程度

**分级**（见Note 20）：

| 敏感度 | 典型用户 | 可接受的数据类型 | 产品策略 |
|-------|---------|----------------|---------|
| **极高** | 医生/律师 | 只能本地部署，数据不出内网 | 提供私有云版本 |
| **高** | 企业高管 | 脱敏后可上传 | 提供数据加密、匿名化 |
| **中** | 普通职场人 | 工作文档可上传 | 标准云服务 |
| **低** | 学生/自媒体 | 无所谓，随便用 | 公有云，数据可用于训练 |

**画像示例**：
```
李明的隐私敏感度：中
  ├─ 表现："公司内部代码不能上传，个人练习代码可以"
  ├─ 行为：用AI时会先删除敏感信息（API key、数据库密码）
  ├─ 痛点："不确定哪些信息敏感，删除太多影响AI理解，删少了担心泄密"
  └─ 产品策略：提供"敏感信息自动检测"（高亮API key、密码等）
```

#### 要素5：多场景切换

**定义**：同一用户在不同场景下对AI的需求完全不同

**AI产品特殊性**：传统软件场景固定（Photoshop=修图），AI产品场景多变（工作写代码 vs 学习看论文 vs 娱乐闲聊）

**画像示例**：
```
李明的多场景需求：

场景1：工作编程（周一-五 9:00-22:00）⭐ 主场景
  ├─ 需求：快速生成代码、代码解释、bug调试
  ├─ 期望：速度>准确性（快速起草即可）
  ├─ 付费意愿：极高（¥299/月，公司报销）
  └─ 设备：MacBook Pro

场景2：个人学习（周末 14:00-17:00）
  ├─ 需求：技术概念讲解、论文翻译
  ├─ 期望：深度>速度（要讲透）
  ├─ 付费意愿：中（¥49/月，自费）
  └─ 设备：iPad

场景3：生活娱乐（晚上 23:00-24:00）
  ├─ 需求：闲聊、讲笑话、写诗
  ├─ 期望：有趣>实用
  ├─ 付费意愿：无（免费就行）
  └─ 设备：iPhone

产品策略：
  → 推出"场景切换"功能（工作模式/学习模式/娱乐模式）
  → 每个模式UI/功能/语气不同
  → 定价：工作模式¥299/月，学习模式¥49/月，娱乐模式免费
```

## 8. 常见错误与陷阱

### 8.1 错误1：完美用户陷阱

**表现**：画像没有缺点、痛点、挫折，完全正面

❌ **错误示例**：
```
李明，28岁程序员
目标：用AI提升效率
行为：每天用豆包50次
态度：非常喜欢AI

问题：
  → 没有痛点（真实用户一定有痛点）
  → 没有挫折（真实用户一定遇到过问题）
  → 没有负面情绪（真实用户有喜怒哀乐）
  → 无法指导产品改进（不知道该解决什么问题）
```

✅ **正确示例**：
```
李明，28岁程序员
目标：用AI提升效率
痛点：AI代码有bug，花2小时debug
挫折时刻："上周AI给了我200行代码，结果有3个bug，
         我直接崩溃了，当时想删掉豆包。"
负面情绪："有时候我很焦虑，不知道AI哪些地方靠谱，
         哪些不靠谱，只能全部检查，很累。"

价值：这些痛点直接指导产品开发"代码解释器""风险评分"功能
```

### 8.2 错误2：虚构画像（非数据驱动）

**表现**：画像完全靠PM拍脑袋，没有真实用户支撑

❌ **错误流程**：
```
Step 1：PM想象"我们的用户应该是这样的"
Step 2：画像诞生（0个真实用户访谈）
Step 3：团队按画像开发产品
Step 4：上线后发现真实用户完全不同 → 失败

案例：某AI产品PM虚构"张三，25岁运营，每天用AI写文案"
     实际数据：真实用户主要是程序员（60%），运营只占5%
     结果：为运营开发的功能使用率<2%，浪费3个月
```

✅ **正确流程**：
```
Step 1：访谈20-30个真实用户
Step 2：聚类分析，找出3-5个模式
Step 3：每个模式综合3-5个真实用户，创建画像
Step 4：用定量数据验证画像假设
Step 5：定期更新（每季度回访用户）

结论：画像必须基于真实数据，不能拍脑袋
```

### 8.3 错误3：画像过多

**表现**：创建10+个画像，团队记不住

❌ **问题**：
- 太多画像 → 团队记不住 → 形同虚设
- 决策时："这个功能对应哪个画像？"→ 无人能答

**数据**：
```
画像数量与使用频率的关系（某AI公司内部研究）

画像数量    团队记住率    决策中使用率    实际价值
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
3个         95%           80%            ⭐⭐⭐⭐⭐
5个         75%           60%            ⭐⭐⭐⭐
8个         40%           25%            ⭐⭐
12个        15%           <5%            ⭐ (形同虚设)

结论：3-5个画像是最优数量
```

✅ **正确做法**：
- 核心画像：3个（覆盖80%用户）
- 边缘画像：最多2个（覆盖特殊场景）
- 总计：不超过5个

### 8.4 错误4：画像创建后不更新

**表现**：2年前创建的画像，从未更新，已与现实脱节

**问题**：
- AI产品用户变化快（6个月一个代际）
- 2024年的"AI新手"，2025年可能已是"AI熟手"
- 旧画像 → 误导决策 → 产品失败

**案例**：
```
2024年Q1：豆包创建"李明"画像
  特征：AI成熟度L2（学徒），不会写复杂Prompt

2025年Q4：李明已进化
  特征：AI成熟度L3（熟手），会写结构化Prompt
         痛点已从"不会用AI"变成"AI不够强大"

如果画像不更新：
  → 产品还在开发"新手引导"功能
  → 但真实用户已不需要，需要"高级功能"
  → 流失到竞品
```

✅ **正确做法**：
- **每季度审查**：回访10个画像对应的真实用户，检查是否还符合
- **每半年更新**：根据最新数据，更新画像的痛点/行为/技能水平
- **记录变化**：在画像文档里记录"2024 Q1 vs 2025 Q1变化"

D. 王总，45岁CEO，引用金句"AI是我的智囊团"

<details>
<summary>点击查看答案</summary>

**答案：B**

**解析**：
- B是**细分市场描述**，不是用户画像
- 问题：
  1. "用户A"不是真实姓名（无法共情）
  2. "25-35岁"太宽泛（应该具体年龄）
  3. "希望工作更高效"是废话（无法指导产品）

好的画像应该：
  ✅ 有真实姓名（李明/小雅/王总）
  ✅ 具体年龄（28岁，不是25-35岁）
  ✅ 可操作的目标/痛点（"论文要读2天"比"希望更高效"具体）
  ✅ 引用金句（用户原话）

</details>

**Q2. 用户画像的核心价值是？**

A. 让产品文档看起来很专业
B. 帮助团队共情用户，做出更好的决策
C. 满足投资人的要求
D. 方便做市场调研报告

<details>
<summary>点击查看答案</summary>

**答案：B**

**解析**：
- 用户画像的核心价值 = **帮助团队共情用户，做出更好的决策**

具体体现：
1. **建立共识**：所有人对"用户"理解一致（李明 vs 模糊的"程序员"）
2. **引导决策**："李明会需要这个功能吗？"（而非"我觉得..."）
3. **激发共情**：想象李明加班到10点的样子（vs 冷冰冰的数据）
4. **提高效率**：决策时间从18天降到9天

为什么其他选项错？
- A：画像不是为了"看起来专业"，是为了指导决策
- C/D：这些是次要价值，不是核心价值

</details>

**Q3. 创建画像时，访谈多少个用户最合适？**

A. 5个（太少了）
B. 10个（勉强够）
C. 20-30个（最佳）
D. 100个（太多了）

<details>
<summary>点击查看答案</summary>

**答案：C**

**解析**：
- **20-30个用户**是最佳数量

原因：
1. **<10个**：样本太小，可能遗漏重要模式
2. **20-30个**：✅ 足够识别3-5个典型模式
             ✅ 每个模式有5+个真实用户支撑
             ✅ 时间成本可控（2-3周完成）
3. **>50个**：边际收益递减（新信息很少），时间成本高

实战数据（某AI公司经验）：
```
访谈用户数    发现新模式概率    时间成本
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
10个          80%              1周
20个          95%              2周  ✅ 最优
30个          98%              3周
50个          99%              5周  (边际收益小)
```

</details>

### 11.2 简答题

**Q4. 豆包的"李明"画像和Kimi的"陈博士"画像，哪个更好？为什么？**

<details>
<summary>点击查看答案</summary>

**答案**：没有绝对的"更好"，取决于产品策略

**豆包的"李明"画像（横向策略）**：

优点：
✅ 覆盖面广（李明+小雅+王总=3个画像覆盖80%用户）
✅ 适合大流量平台（字节系有抖音/今日头条导流）
✅ 风险分散（一个市场失败，其他市场补）

缺点：
❌ 深度不够（李明画像1页，细节有限）
❌ 差异化难（与竞品功能类似）
❌ 用户期望复杂（3类用户需求各异，难以全满足）

适用场景：
→ 大厂，资源充足，追求规模

**Kimi的"陈博士"画像（垂直策略）**：

优点：
✅ 极致深度（陈博士画像5页，包含"典型一周"细节）
✅ 差异化明显（"20万字超长文本"是独特卖点）
✅ 高溢价能力（学术版¥149/月，付费率18%）
✅ 护城河深（竞争对手短期无法复制）

缺点：
❌ 市场规模小（学术+法律+咨询 < 1000万用户）
❌ 天花板明显（垂直场景增长有限）
❌ 依赖单一能力（如果竞争对手追上"长文本"，优势消失）

适用场景：
→ 创业公司，技术有独特优势，追求高利润率

**结论**：
- **豆包策略**：适合"有流量、有资源、追求规模"的大厂
- **Kimi策略**：适合"有技术壁垒、追求利润率"的创业公司

两种策略都成功，关键是**画像与产品策略匹配**。

</details>

**Q5. 你的AI产品团队创建了5个用户画像，但3个月后发现团队决策时从不提及画像，形同虚设。可能是什么原因？如何改进？**

<details>
<summary>点击查看答案</summary>

**答案**：

**可能原因**：

1. **画像太抽象，难以共情**
   - 问题：只有冷冰冰的数据（"28岁程序员"），没有故事/金句
   - 改进：添加"引用金句""挫折时刻""一天的生活"

2. **画像太多，记不住**
   - 问题：5个画像，团队记不住每个画像的特点
   - 改进：减少到3个核心画像，贴海报在办公室

3. **画像与真实用户脱节**
   - 问题：画像是PM拍脑袋虚构的，非真实数据
   - 改进：回访真实用户，用数据验证画像

4. **画像创建后没有传播**
   - 问题：画像文档躺在Notion里，新人不知道
   - 改进：新人入职第一天培训画像，会议上主动引用

5. **领导不使用画像**
   - 问题：CEO/VP开会时不提画像，团队也不敢提
   - 改进：高层带头使用（"李明会需要这个功能吗？"）

6. **画像没有融入决策流程**
   - 问题：PRD模板、会议议程里没有"画像评估"环节
   - 改进：制度化要求：每个功能需求必须标注"针对哪个画像"

**改进行动计划**：

Week 1：
✅ 回访10个用户，验证画像是否准确
✅ 添加"引用金句""挫折时刻"让画像更生动

Week 2：
✅ 制作1页纸画像海报，贴在办公室
✅ 录制5分钟画像讲解视频，发全员

Week 3：
✅ 修改PRD模板，增加"目标画像"字段
✅ 产品会议议程增加"画像评估"环节

Week 4：
✅ CEO/VP带头在会议上引用画像
✅ 每周复盘：本周决策中画像使用率（目标>50%）

Month 3：
✅ 检查画像使用率是否提升到60%+
✅ 如果仍低，考虑重新创建画像（可能是画像本身有问题）

</details>

### 练习1：为你的AI产品创建第一个画像

**任务**：
