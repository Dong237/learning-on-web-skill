| 交付物 | 必须有 | 适用场景 | 耗时 | 受众 |
|--------|--------|---------|------|------|
| **PRD** | ✅ | 所有新功能 | 1-3天 | 工程师/设计师/测试 |
| **User Story** | ✅ | 敏捷开发 | 1-2天 | 开发团队 |
| **Roadmap** | ✅ | 季度规划 | 2-5天 | 老板/团队/客户 |
| **原型图** | ✅ | 有UI变化的功能 | 2-7天 | 设计师/开发 |
| **竞品分析** | ⭕ | 新产品/重大功能 | 3-7天 | 老板/团队 |
| **数据埋点** | ✅ | 需要数据验证的功能 | 0.5-1天 | 数据/开发 |
| **测试用例** | ⭕ | 复杂功能 | 1-2天 | 测试工程师 |

**说明**:
- ✅ 必须有:不写会影响开发质量
- ⭕ 推荐有:提升决策质量,但非必须

## PRD产品需求文档

### PRD的本质:回答5个核心问题

```
1. Why? (为什么做)
   → 背景、目标、用户价值

2. What? (做什么)
   → 功能清单、优先级

3. Who? (给谁用)
   → 目标用户、使用场景

4. How? (怎么做)
   → 功能流程、交互逻辑

5. When? (何时交付)
   → 里程碑、验收标准
```

### PRD标准模板(2026中国版)

```markdown
# 产品需求文档(PRD)

## 文档信息
- 项目名称:AI智能推荐问题
- 产品:豆包AI助手
- 版本:V1.0
- 作者:张三(PM)
- 创建日期:2026-01-15
- 评审日期:2026-01-18
- 开发周期:2 Sprints(4周)

---

## 一、需求背景 (Background)

### 1.1 问题陈述
用户在使用AI对话时,面对空白输入框不知道问什么,导致:
- 首次使用激活率仅30%(行业平均45%)
- 新用户7日留存38%(目标50%)
- 用户调研:"不知道AI能帮我做什么"(15/20用户提及)

### 1.2 商业目标
- 提升首次使用激活率:30% → 50%
- 提升新用户7日留存:38% → 50%
- 预计增加DAU:+15% (从100万到115万)

### 1.3 用户价值
- 降低使用门槛(新手快速上手)
- 发现AI能力(了解产品价值)
- 节省思考时间(直接点击推荐)

---

## 二、目标用户 (Target Users)

### 2.1 用户画像
**主要用户**:新注册用户(0-7天)
- 年龄:22-35岁
- 职业:学生、上班族
- 场景:首次打开APP,不知道从何开始

**次要用户**:低频用户(月活<4次)
- 场景:偶尔想起来用,但想不到用来干嘛

### 2.2 用户旅程
1. 打开APP → 看到空白输入框 → 不知道说什么
2. 看到推荐问题("帮我写周报") → 点击
3. AI回答 → 满意 → 继续尝试其他推荐
4. 形成使用习惯 → 7日内回访

---

## 三、功能需求 (Functional Requirements)

### 3.1 功能列表

| 功能ID | 功能名称 | 优先级 | 说明 |
|--------|---------|--------|------|
| F1 | 智能推荐问题 | P0 | 根据用户画像推荐3个问题 |
| F2 | 点击直接提问 | P0 | 点击推荐问题,自动填入输入框 |
| F3 | 推荐刷新 | P1 | 不喜欢当前推荐,刷新换一批 |
| F4 | 推荐个性化 | P2 | 基于历史对话,推荐相关问题 |

**优先级说明**:
- P0:MVP必须有,不做无法上线
- P1:重要但非紧急,第二期做
- P2:锦上添花,资源充足再做

### 3.2 功能详细设计

**功能F1:智能推荐问题**

**触发时机**:
- 新用户首次打开APP
- 用户清空输入框后闲置3秒
- 用户主动点击"试试问我"按钮

**推荐策略**:
```
IF 新用户(注册<7天)
  THEN 推荐通用高频问题(基于全量用户数据)

IF 老用户(注册>7天)
  THEN 推荐个性化问题(基于用户历史对话)

推荐问题池(V1.0):
- 工作场景(30%):帮我写周报、总结会议纪要、起草邮件
- 学习场景(30%):解释XX概念、推荐学习路径、生成练习题
- 生活场景(20%):制定健身计划、推荐菜谱、旅游攻略
- 创意场景(20%):起个项目名、写个slogan、编个故事
```

**UI展示**:
```
┌─────────────────────────────────────┐
│  豆包AI助手                          │
├─────────────────────────────────────┤
│                                     │
│  👋 你好!我能帮你:                  │
│                                     │
│  💼 帮我写本周工作总结  →            │
│  📚 解释"产品市场契合"这个概念  →    │
│  🍳 推荐3道30分钟内能做的晚餐  →     │
│                                     │
│  [换一批]                           │
│                                     │
│  ─────────────────────────           │
│                                     │
│  [输入你的问题...]                   │
│                                     │
└─────────────────────────────────────┘
```

**交互逻辑**:
1. 用户点击推荐问题
2. 问题自动填入输入框
3. 自动发送(无需点击发送按钮)
4. AI开始回答

**边界情况**:
- 如果AI推荐失败(接口超时):显示默认3个问题
- 如果用户点击"换一批"超过3次:提示"直接输入你的问题吧"
- 如果用户开始输入:隐藏推荐问题

## 四、非功能需求 (Non-Functional Requirements)

### 4.1 性能要求
- 推荐问题加载时间:<1秒(P0)
- AI回答首token生成:<2秒(P0)
- 接口成功率:>99%(P0)

### 4.2 兼容性要求
- iOS:14.0+
- Android:9.0+
- 微信小程序:兼容(P1)

### 4.3 数据安全
- 推荐问题不涉及敏感信息
- 用户历史对话加密存储
- 符合《个人信息保护法》
```

```
## 九、FAQ (常见问题)

**Q1:为什么是3个推荐,不是5个?**
A:参考竞品(ChatGPT/Kimi)都是3个。3个不多不少,选择压力小。AB测试后可调整。

**Q2:推荐问题多久更新一次?**
A:V1.0每周人工更新,V2.0引入AI自动生成(基于热门对话)

**Q3:如果用户从来不点推荐怎么办?**
A:监控数据,如点击率<10%,说明推荐无价值,下线功能。
```

### PRD的常见错误与修正

**错误1:只写"做什么",不写"为什么"**

```
❌ 错误PRD:
需求:增加语音输入功能
功能:点击麦克风按钮,说话,转文字

✅ 正确PRD:
背景:用户调研显示,25%用户在开车/做饭时想用AI但无法打字
目标:提升使用场景覆盖率,预计DAU+10%
功能:语音输入(详细流程...)
```

**错误2:需求描述模糊**

```
❌ "优化用户体验" (什么叫优化?怎么衡量?)
✅ "将首次对话完成率从60%提升到75%,通过增加新手引导实现"

❌ "AI回答要快" (多快算快?)
✅ "AI首token生成时间<2秒,90分位延迟<5秒"
```

**错误3:缺少验收标准**

```
❌ 只写"功能完成即可"
✅ 明确DoD:
- 单元测试覆盖率>80%
- PM验收通过
- 无P0 Bug
- 埋点数据正常
- 3个设备测试通过
```

## User Story用户故事

### User Story的格式

```
作为[角色],我希望[功能],以便[价值]

As a [role], I want [feature], so that [benefit]
```

### 优秀User Story的INVEST原则

| 原则 | 含义 | 示例 |
|------|------|------|
| **I**ndependent | 独立 | Story之间不强依赖,可单独开发 |
| **N**egotiable | 可协商 | 不是合同,可与团队讨论细节 |
| **V**aluable | 有价值 | 每个Story对用户有明确价值 |
| **E**stimable | 可估算 | 团队能估算工作量(Story Point) |
| **S**mall | 小而美 | 1个Sprint内能完成 |
| **T**estable | 可测试 | 有明确的验收标准 |

### User Story模板与示例

**模板**:

```markdown
## User Story: [简短标题]

**作为**:[角色]
**我希望**:[功能]
**以便**:[价值]

### 验收标准(Acceptance Criteria)
- [ ] 场景1:Given [前置条件], When [操作], Then [结果]
- [ ] 场景2:...

### Story Point估算
- 工作量:[X] SP (1 SP ≈ 0.5天)
- 复杂度:[低/中/高]

### 技术依赖
- 依赖Story:无 / [Story ID]
- 依赖系统:XX API / XX数据库

### UI/UX
- 原型链接:[Figma link]
- 交互说明:[简述]
```

**实战示例(AI对话产品)**:

```markdown
## User Story #42: 智能推荐问题

**作为**:新注册用户
**我希望**:打开APP时看到推荐问题
**以便**:快速了解AI能做什么,降低首次使用门槛

### 验收标准
- [ ] Given 新用户首次打开APP, When 进入首页, Then 显示3个推荐问题
- [ ] Given 推荐问题已显示, When 点击任意推荐, Then 问题自动发送给AI
- [ ] Given 推荐问题已显示, When 点击"换一批", Then 显示新的3个问题(不重复)
- [ ] Given 用户开始输入, When 输入框有文字, Then 隐藏推荐问题

### Story Point估算
- 工作量:5 SP (前端2 SP + 后端3 SP)
- 复杂度:中(需调用AI推荐接口)

### 技术依赖
- 依赖Story:无
- 依赖系统:豆包AI推荐API(已就绪)

### UI/UX
- 原型链接:[Figma设计稿](link)
- 交互说明:点击推荐后自动发送,无需确认
```

### User Story拆分技巧

**反模式:Story太大**

```
❌ 超大Story:
"作为用户,我希望有完整的AI写作功能,以便提升工作效率"
→ 太大,无法在1个Sprint完成

✅ 拆分成小Story:
Story 1:"作为用户,我希望输入主题,AI生成大纲"
Story 2:"作为用户,我希望基于大纲,AI扩写每个章节"
Story 3:"作为用户,我希望AI润色文章,改进表达"
Story 4:"作为用户,我希望导出为Word/PDF"
```

**拆分方法**:
1. **按流程拆**:注册 → 登录 → 使用
2. **按优先级拆**:P0核心功能 → P1增强功能
3. **按角色拆**:新用户 → 老用户 → 付费用户
4. **按平台拆**:Web → iOS → Android

## 💡 实战洞察 (AI PM Practitioner Insights)

> **洞察1:文档不是目的,沟通才是**
>
> 很多初级PM陷入"文档陷阱":
> - 花3天写了50页PRD,没人看
> - Roadmap做得很漂亮,但团队不理解战略
> - User Story写得规范,但工程师还是不懂需求

>
> **高效做法**:
> - PRD控制在10页内,核心内容前3页说完
> - Roadmap开评审会,讲给团队听(PPT形式)
> - User Story当面讲,文档只是备忘录
>
> **字节文化**:"文档即代码"≠长篇大论,而是"清晰+简洁+可执行"

> **洞察2:AI时代,PM要会"Prompt工程"**
>
> 2026年,PM的核心技能变了:
> - 2020年:会写文档(Word/PPT)
> - 2026年:会写Prompt(让AI帮你写文档)

> - 我用通义千问生成PRD草稿:30分钟 vs 传统3小时

## ✅ 自测题 (Self-Check Questions)

<details>
<summary>Q1: PRD中最容易被忽略但最重要的部分是什么?</summary>

**答案**:"为什么做"(Why)部分

**很多PM犯的错误**:
- 只写"做什么"(功能清单)
- 忽略"为什么做"(背景、目标、价值)

**后果**:
- 工程师不理解需求,执行偏差
- 老板不知道为什么批这个项目
- 后续迭代时,忘了初衷,方向跑偏

**正确做法**:
PRD的前3页必须回答:
1. **问题是什么**?(用数据说话)
2. **为什么现在做**?(时机/机会)
3. **目标是什么**?(可量化的成功指标)
4. **用户价值是什么**?(对用户有什么好处)

**案例**:
某PM只写"增加语音输入功能",团队开发后发现:
- 用户根本不用(使用率<1%)
- 原因:PM没说清"为什么做"(解决什么场景的问题)

如果PRD写清楚:
"用户在开车/做饭时想用AI但无法打字,25%用户有此需求"
→ 团队会针对性优化(如增加唤醒词,免提操作)

</details>

<details>
<summary>Q2: Roadmap给不同受众看,内容应该有什么差异?</summary>

**答案**:一份Roadmap,三个版本

**版本1:给老板看(战略版)**
- 重点:商业目标、ROI、市场竞争
- 形式:PPT,10页以内
- 内容:
  - Q1-Q4大方向(不要太细节)
  - 每个季度的核心目标(用户增长/营收)
  - 资源需求(人力/预算)
  - 风险与依赖

**版本2:给团队看(执行版)**
- 重点:具体功能、时间节点、优先级
- 形式:飞书多维表格 / Jira Roadmap
- 内容:
  - 每个Sprint的User Story
  - 技术依赖(需要哪个团队配合)
  - DoD(什么算完成)

**版本3:给客户/用户看(营销版)**
- 重点:新功能、用户价值、上线时间
- 形式:产品官网 / 公众号
- 内容:
  - "即将上线"功能(制造期待)
  - 用户能获得什么好处
  - Beta测试招募

**实战案例**:
字节豆包的Roadmap:
- 给张一鸣(老板):H1 DAU目标200万,商业化路径
- 给研发团队:Q1 Sprint 1-6详细任务
- 给用户:官网"即将上线:语音输入、文档上传"

</details>

<details>
<summary>Q3: User Story的Story Point应该由谁估算?PM还是工程师?</summary>

**答案**:必须由工程师估算,PM只能参与讨论

**原因**:
- Story Point反映的是**工作量+复杂度**
- 只有实际干活的工程师知道要花多少时间
- PM估算往往过于乐观(低估难度)

**正确流程**:

1. **PM讲解Story**
   - 背景、需求、验收标准
   - 澄清工程师的疑问

2. **团队集体估算**(Planning Poker)
   - 每个工程师私下选一个数字(1/2/3/5/8/13)
   - 同时亮牌
   - 如果差异大,讨论为什么(最大和最小解释理由)
   - 再次投票,直到收敛

3. **PM记录共识**
   - 团队一致认为是5 SP
   - 如果无法收敛,拆分Story

**反例(PM强行估算)**:
PM:"我觉得这个功能很简单,2 SP就够了"
工程师:"你不懂技术,这需要重构数据库,至少8 SP"
→ 结果:冲突,团队士气低落

**正例(工程师估算)**:
PM:"这个Story大概多少SP?"
工程师A:"我觉得5 SP"
工程师B:"我觉得8 SP,因为要适配iOS/Android"
团队讨论 → 一致认为6 SP
→ 结果:共识,团队承诺

</details>

<details>
<summary>Q4: 如何判断一份PRD是"好"还是"坏"?</summary>

**好PRD的5个标准**:

**1. 清晰性(Clarity)**
- 工程师/设计师读完,100%理解要做什么
- 无歧义(不需要反复找PM确认)

测试方法:让一个新人读,能否复述需求?

**2. 完整性(Completeness)**
- Why(背景+目标)
- What(功能清单)
- Who(用户)
- How(流程+交互)
- When(时间+验收标准)

测试方法:有没有漏掉关键信息?

**3. 可执行性(Actionable)**
- 技术方案可行(不是天马行空)
- 资源明确(人力/时间/预算)
- 验收标准清晰(什么算"做完")

测试方法:工程师读完能直接开发吗?

**4. 简洁性(Concise)**
- 核心内容10页内说完
- 无废话(不要为了凑字数写PRD)

测试方法:删掉20%内容,影响理解吗?

**5. 数据驱动(Data-Driven)**
- 有用户调研数据支撑
- 有竞品分析对比
- 有可量化的目标

测试方法:PRD里有具体数字吗?(不是"很多用户",而是"25%用户")

**坏PRD的典型特征**:
- ❌ 只有功能列表,没有Why
- ❌ 模糊描述("优化体验""提升性能")
- ❌ 缺少验收标准(不知道何时算完)
- ❌ 没有优先级(P0/P1不分)
- ❌ 超过30页(废话太多)

</details>
