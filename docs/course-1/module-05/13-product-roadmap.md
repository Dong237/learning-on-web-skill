## 1. 产品路线图的本质

### 1.1 什么是产品路线图?

**产品路线图 (Product Roadmap)** = 产品愿景到执行的桥梁

```
愿景 (Vision)
"3年后成为中国第一的AI办公助手"
         ↓
路线图 (Roadmap)
"Q1: 文档AI / Q2: 会议AI / Q3: 邮件AI / Q4: 数据AI"
         ↓
执行 (Execution)
"本周sprint: 实现文档摘要功能 (User Story #1234)"
```

**路线图 ≠ 项目计划**

| 维度 | 产品路线图 (Roadmap) | 项目计划 (Project Plan) |
|------|---------------------|----------------------|
| **时间粒度** | 季度/月 (3-18个月) | 周/天 (1-3个月) |
| **关注重点** | Why (为什么做) + What (做什么) | How (怎么做) + When (什么时候) |
| **灵活性** | 高 (可调整优先级) | 低 (需按计划执行) |
| **受众** | 全公司 + 客户 | 产品/研发团队 |
| **粒度** | 功能主题 (如 "AI写作助手") | 具体任务 (如 "实现GPT-4接口") |

### 1.2 产品路线图的4大作用

#### 作用1: 对内 - 对齐团队方向 (Alignment)

**问题场景**: 研发说 "我们应该先做性能优化", 运营说 "我们需要新功能拉新", 销售说 "客户要的功能还没做"

**路线图解决**:
```
Q1优先级 (全公司达成共识):
1️⃣ 稳定性优化 (P0 - 必须做)
   → 研发满意 ✅
2️⃣ AI写作功能 (P0 - 销售承诺的)
   → 销售满意 ✅
3️⃣ 社交分享功能 (P1 - Q2再做)
   → 运营暂时接受 ⚠️
```

---

#### 作用2: 对外 - 管理客户期待 (Expectation Management)

**错误做法**: 销售承诺客户 "下个月上线XX功能", 结果研发延期, 客户流失

**正确做法**: 路线图公开, 客户自己看到 "该功能在Q3", 提前调整预期

**2026中国AI产品案例** (飞书AI):
- 公开路线图: "2026 H1规划 - 文档AI / 会议AI / 邮件AI"
- 客户看到路线图后: "邮件AI在H1, 我可以等, 先用文档AI"
- 避免流失 + 提升信任

---

#### 作用3: 战略 - 资源分配依据 (Resource Allocation)

**场景**: CEO问 "为什么AI团队有20人, 但数据团队只有5人?"

**路线图回答**: "Q1-Q2我们主攻AI写作 (核心战略), 需要AI算法+NLP团队; Q3才做数据分析, 所以数据团队暂时不扩招."

---

#### 作用4: 决策 - 说 "No" 的依据 (Prioritization)

**场景**: 客户提需求 "能不能加个视频通话功能?"

**没有路线图**: PM不知道该不该做, 拖延决策
**有路线图**: "我们未来18个月聚焦AI写作, 视频通话不在规划内, 暂时不做."

## 2. 路线图的3种经典类型

### 2.1 类型1: 时间线式路线图 (Timeline Roadmap)

**适用场景**:
- 产品成熟, 需求稳定
- 需要向外部 (投资人/客户) 展示明确计划
- 硬件产品 (时间节点固定)

**优点**: 清晰直观, 容易理解
**缺点**: 灵活性差, 容易变成 "承诺清单"

**示例: 某AI办公助手2026路线图**

```
时间线式路线图 (Timeline-based)

Q1 2026           Q2 2026           Q3 2026           Q4 2026
────────────────────────────────────────────────────────────→
│                 │                 │                 │
│ 📄 文档AI       │ 📊 数据AI       │ 📧 邮件AI       │ 🎙️ 会议AI
│ - AI摘要        │ - 智能图表      │ - 智能回复      │ - 实时转写
│ - 智能改写      │ - 数据问答      │ - 邮件分类      │ - 会议纪要
│ - 多语言翻译    │ - 趋势预测      │ - 日程提取      │ - 行动项提取
│                 │                 │                 │
└─ Beta测试       └─ 企业版上线    └─ API开放       └─ 多模态升级
   (1000用户)        (目标100家)      (开发者生态)      (图+音+视频)
```

**2026中国AI产品实例** (通义千问企业版):
- Q1: 知识库问答 (企业文档检索)
- Q2: 智能客服 (7×24小时)
- Q3: 数据分析助手 (对接BI系统)
- Q4: 行业定制模型 (医疗/金融/教育)

---

### 2.2 类型2: 主题式路线图 (Theme-based Roadmap)

**适用场景**:
- 创业公司, 需求快速变化
- 敏捷团队, 不想被时间绑死
- 内部沟通为主

**优点**: 灵活, 可随时调整优先级
**缺点**: 没有时间承诺, 外部客户可能不满意

**示例: 某AI写作助手主题式路线图**

```
主题式路线图 (Theme-based)

┌──────────────────────────────────────────┐
│ 主题1: 核心写作能力 (P0 - 必须做)         │
├──────────────────────────────────────────┤
│ ✅ 完成: 周报生成 / 会议纪要 / 邮件回复  │
│ ⏳ 进行中: 小红书文案 / 广告语           │
│ 📋 待做: 长文章生成 / 论文润色           │
└──────────────────────────────────────────┘

┌──────────────────────────────────────────┐
│ 主题2: 用户体验优化 (P1 - 重要)          │
├──────────────────────────────────────────┤
│ ✅ 完成: Markdown编辑器 / 模板库         │
│ ⏳ 进行中: 一键复制 / 历史记录           │
│ 📋 待做: 协作编辑 / 版本对比             │
└──────────────────────────────────────────┘

┌──────────────────────────────────────────┐
│ 主题3: 商业化 (P1 - 重要)                │
├──────────────────────────────────────────┤
│ ✅ 完成: 会员体系 / 支付接入             │
│ ⏳ 进行中: 企业版 / API接口              │
│ 📋 待做: 分销体系 / 私有化部署           │
└──────────────────────────────────────────┘

┌──────────────────────────────────────────┐
│ 主题4: 技术债务 (P2 - 可选)              │
├──────────────────────────────────────────┤
│ ⏳ 进行中: 性能优化 / 代码重构           │
│ 📋 待做: 单元测试覆盖率提升              │
└──────────────────────────────────────────┘
```

**优势**: 清晰展示优先级, 可以随时调整各主题的进度

---

### 2.3 类型3: Now-Next-Later 路线图 (敏捷路线图)

**适用场景**:
- 快速迭代的产品 (AI产品首选!)
- 市场变化快, 难以预测6个月后
- 需要灵活调整方向

**优点**: 最灵活, 不承诺具体时间
**缺点**: 需要团队高度自律, 否则变成 "没有计划"

**示例: 某AI产品Now-Next-Later路线图**

```
Now-Next-Later路线图

┌─────────────────────────────────────────┐
│ 🚀 NOW (当前sprint, 2周)                │
├─────────────────────────────────────────┤
│ ✅ AI摘要功能优化 (准确率提升10%)       │
│ ⏳ 新用户引导流程 (提升激活率)          │
│ ⏳ Bug修复 (崩溃率降到0.1%)             │
└─────────────────────────────────────────┘

┌─────────────────────────────────────────┐
│ 📅 NEXT (下1-2个sprint, 1个月)          │
├─────────────────────────────────────────┤
│ 📋 多语言翻译 (支持中英日韩)            │
│ 📋 文档上传 (PDF/Word解析)              │
│ 📋 会员体系 (付费功能分级)              │
└─────────────────────────────────────────┘

┌─────────────────────────────────────────┐
│ 🔮 LATER (未来2-6个月, 待验证)          │
├─────────────────────────────────────────┤
│ 💡 语音输入 (多模态交互)                │
│ 💡 企业版 (团队协作)                    │
│ 💡 API开放 (开发者生态)                 │
│ 💡 行业定制 (医疗/法律/教育)            │
└─────────────────────────────────────────┘
```

**2026中国AI产品最爱**: 豆包/Kimi等快速迭代产品都用这种模式

**为什么AI产品适合Now-Next-Later?**
- AI技术更新快 (3个月前的GPT-4 vs 现在的GPT-4.5, 能力差距巨大)
- 用户需求变化快 (今天流行长文本, 明天可能流行多模态)
- 竞争激烈, 需要快速响应

## 3. 如何制定有效的产品路线图

### 3.1 制定路线图的6步法

```
第1步: 明确产品愿景 (Vision)
"3年后我们要成为什么?"

第2步: 分解年度目标 (Annual Goals)
"今年必须达成的3个里程碑是什么?"

第3步: 识别关键主题 (Themes)
"为了达成目标, 需要做哪些方向的工作?"

第4步: 收集需求池 (Feature Backlog)
"所有可能做的功能列表"

第5步: 优先级排序 (Prioritization)
"哪些是P0必须做, 哪些是P2可以不做?"

第6步: 分配到时间段 (Timeline Allocation)
"Q1做什么, Q2做什么?"
```

### 3.2 好路线图的5个特征 (SMART原则改编)

| 特征 | 说明 | 反例 (坏路线图) | 正例 (好路线图) |
|------|------|----------------|----------------|
| **Strategic** (战略驱动) | 每个功能都和战略目标对齐 | "Q1做10个功能" (功能清单) | "Q1主攻企业市场, 做团队协作+权限管理" |
| **Measurable** (可衡量) | 每个里程碑有明确成功指标 | "Q2提升用户体验" (模糊) | "Q2激活率从20%提升到35%" |
| **Achievable** (可实现) | 资源匹配, 不画大饼 | "Q1做100个功能" (团队只有5人) | "Q1做3个核心功能, 打磨体验" |
| **Relevant** (相关性) | 和目标用户需求匹配 | "做区块链功能" (用户不需要) | "做AI摘要" (用户核心痛点) |
| **Timebound** (有时间框) | 明确时间范围 (但允许调整) | "未来做XX" (无时间概念) | "Q3-Q4做企业版" |

### 3.3 路线图制定实战案例

**背景**: 某AI客服创业公司, 2026年Q1刚完成天使轮融资, 需要制定全年路线图.

**第1步: 产品愿景**
> "成为中国中小企业首选的AI客服解决方案, 2年内服务1万家企业"

**第2步: 年度目标** (2026)
1. 签约100家付费企业客户 (ARR达到500万)
2. 产品稳定性达到99.9% (SLA指标)
3. 完成Pre-A轮融资

**第3步: 关键主题**
- 主题1: 产品稳定性 (支撑企业级SLA)
- 主题2: 功能完善 (满足中小企业需求)
- 主题3: 销售赋能 (让销售好卖)
- 主题4: 品牌建设 (提升知名度)

**第4步: 需求池** (简化版)
- 多渠道接入 (微信/网页/电话)
- 智能问答 (FAQ自动回复)
- 人工客服协作 (AI+人工混合模式)
- 客户数据分析 (洞察客户需求)
- 工单系统 (复杂问题工单化)
- 知识库管理 (客服知识沉淀)
- API接口 (集成到客户系统)
- 私有化部署 (大客户需求)

**第5步: 优先级排序** (RICE模型)

| 功能 | Reach | Impact | Confidence | Effort | RICE分数 | 优先级 |
|------|-------|--------|-----------|--------|---------|--------|
| 智能问答 | 100企业 | 3 (高) | 100% | 4周 | 75 | P0 |
| 多渠道接入 | 100企业 | 3 (高) | 80% | 6周 | 40 | P0 |
| 人工协作 | 100企业 | 2 (中) | 90% | 8周 | 22.5 | P1 |
| 数据分析 | 50企业 | 2 (中) | 70% | 4周 | 17.5 | P1 |
| 工单系统 | 30企业 | 2 (中) | 80% | 6周 | 8 | P2 |
| API接口 | 20企业 | 3 (高) | 60% | 10周 | 3.6 | P2 |
| 私有化部署 | 5企业 | 3 (高) | 50% | 16周 | 0.47 | P3 |

**第6步: 分配到时间段**

```
2026产品路线图 (Now-Next-Later版)

NOW (Q1 2026)
├─ 智能问答优化 (准确率提升到90%)
├─ 微信渠道接入 (覆盖80%中小企业)
└─ 产品稳定性 (可用性99.5% → 99.9%)

NEXT (Q2-Q3 2026)
├─ 人工客服协作 (AI不会的转人工)
├─ 客户数据分析 (帮客户优化服务)
├─ 网页+电话渠道接入 (全渠道覆盖)
└─ 知识库管理 (让客户自己维护FAQ)

LATER (Q4 2026 - 待验证)
├─ 工单系统 (复杂问题跟踪)
├─ API开放 (集成到客户CRM/ERP)
└─ 私有化部署POC (试点1-2家大客户)
```

## 4. AI产品路线图的特殊挑战

### 4.1 挑战1: 技术不确定性高

**问题**: AI模型更新快, 今天规划的功能, 明天可能因为新模型而不需要了

**案例** (2024-2025真实案例):
- 2024年Q1某公司规划 "AI文档摘要功能", 需要自研模型 (预计3个月)
- 2024年Q3豆包开放API, 直接调用即可, 1周搞定
- 结果: 原计划3个月工作量, 浪费了2个月

**应对策略**:
1. **缩短规划周期**: 不做12个月路线图, 只做3-6个月
2. **技术选型灵活**: 优先用API, 不轻易自研
3. **保留调整空间**: 用Now-Next-Later, 不用固定时间线

---

### 4.2 挑战2: 用户需求变化快

**问题**: AI产品教育市场阶段, 用户今天要A, 明天可能改要B

**案例** (Kimi的经验):
- 2024年初: 用户要 "AI聊天"
- 2024年中: 用户要 "AI搜索"
- 2024年底: 用户要 "AI长文本阅读"
- Kimi策略: 快速验证, 发现长文本需求最强, 立刻allin

**应对策略**:
1. **MVP快速验证**: 2周上线最小可用版本, 看数据
2. **数据驱动调整**: 每月review路线图, 根据留存率/活跃度调整
3. **保留20%探索时间**: 团队20%时间做 "未来可能需要" 的探索

---

### 4.3 挑战3: 竞争压力大

**问题**: 竞品功能上线快, 你的路线图可能被打乱

**案例** (2025某AI写作助手):
- 原计划Q2做 "多语言翻译"
- Q1竞品Jasper.ai上线翻译功能, 用户开始流失
- 被迫调整: Q1紧急上线翻译, 原计划的 "模板库" 延后到Q2

**应对策略**:
1. **竞品监控**: 每周review竞品动态, 提前预判
2. **保留应急资源**: 团队20%时间用于 "应急需求"
3. **差异化战略**: 不跟风做 "我也有", 聚焦 "我独有"

## 5. 路线图的沟通与管理

### 5.1 不同受众的路线图版本

**同一个产品, 需要3-4个版本的路线图**

| 受众 | 关注点 | 路线图形式 | 示例 |
|------|--------|-----------|------|
| **CEO/投资人** | 战略方向 + 里程碑 | 高层次主题 + 关键指标 | "Q1企业版上线, Q2签约100家客户, Q3 ARR达500万" |
| **销售团队** | 能卖的功能 + 时间 | 功能清单 + 明确时间 | "Q2上线API接口, 可以卖给开发者客户" |
| **客户** | 承诺的功能 + 进展 | 公开路线图 (保守承诺) | "2026 H1规划: 智能问答 (已上线) / 数据分析 (开发中) / 工单系统 (规划中)" |
| **产品/研发团队** | 详细任务 + 优先级 | Now-Next-Later + User Story | "NOW: US-1234 (AI摘要优化) / NEXT: US-5678 (多语言支持)" |

**实战技巧**:
- ✅ 对外 (客户/投资人): 只承诺70%确定的, 留30%buffer
- ✅ 对内 (团队): 100%透明, 包括 "可能会砍掉" 的功能
- ⚠️ 避免: 给销售看太细节的路线图, 他们会当承诺去卖

---

### 5.2 路线图的Review机制

**建议频率**:
- **每周**: 团队内部review NOW (当前sprint), 调整优先级
- **每月**: 产品/研发/销售三方review NEXT, 看市场反馈是否需要调整
- **每季度**: 全公司review全年路线图, 重大调整需CEO批准

**月度Review模板**:
```
路线图Review会议 (每月最后一周)

参会人: PM / 研发负责人 / 销售负责人 / 运营负责人

议程:
1️⃣ 上月完成情况 (30分钟)
   - 计划完成3个功能, 实际完成2个, 延期1个
   - 延期原因: XX技术难度超预期
   - 数据反馈: 上线功能的留存率/满意度

2️⃣ 本月计划调整 (30分钟)
   - 原计划: A → B → C
   - 市场反馈: 客户强烈要求D功能
   - 调整后: D (紧急插入) → A (降级到下月) → B

3️⃣ 下季度规划 (30分钟)
   - NEXT里的功能是否还合理?
   - 有没有新的机会/威胁需要调整?

4️⃣ Action Items (10分钟)
   - 更新路线图文档
   - 同步给销售/客户
```

### 5.3 如何应对 "路线图变更"

**现实**: 路线图100%会变, 关键是如何管理变更

**变更原则**:
```
变更评估框架:

1️⃣ 紧急度 (Urgency)
   - P0: 影响核心业务 (如线上bug导致客户流失) → 立刻调整
   - P1: 客户强烈要求, 但不影响当前业务 → 下个sprint考虑
   - P2: Nice to have → 加入LATER池, 暂不做

2️⃣ 影响范围 (Impact)
   - 高影响 (如核心功能优化) → 优先级提升
   - 低影响 (如小优化) → 排队等待

3️⃣ 资源成本 (Effort)
   - 低成本 (<1周) → 可以插队
   - 高成本 (>1个月) → 需要CEO批准, 砍掉其他功能

决策矩阵:
紧急度高 + 影响大 + 成本低 → 立刻做
紧急度低 + 影响小 + 成本高 → 不做
其他情况 → 权衡评估
```

**沟通话术** (告诉客户路线图变更):
- ❌ 错误: "抱歉, 我们延期了, 因为技术问题"
- ✅ 正确: "基于用户反馈, 我们调整了优先级, 先做大家更需要的XX功能, 原计划的YY延后到Q3, 感谢理解"

## 实战洞察

### 💡 洞察1: "承诺保守, 交付超预期" - 路线图沟通的黄金法则

**实战故事** (某AI SaaS产品负责人):

> "我们2024年犯过错: 路线图承诺 'Q1上线10个功能', 结果只做了6个, 客户觉得我们 '不靠谱', 续约率下降20%.
>
> 后来调整策略: 路线图只承诺70%确定的, 比如 'Q1上线核心功能A + B', 实际我们规划了A/B/C/D 4个, 但对外只说A和B. 结果:
> - Q1上线A + B (承诺达成) ✅
> - 额外交付C (超预期惊喜) 🎉
> - D因为技术问题延期, 但客户不知道, 所以没有失望
>
> 结果: 客户满意度提升, 续约率回升到85%."

**黄金法则**:
1. **对外承诺 = 70%确定 + 保守时间** (如内部预计2个月, 对外说Q2或3个月)
2. **内部规划 = 120%野心 + 激进时间** (挑战团队极限)
3. **差额 = 惊喜空间** (如果提前完成, 客户觉得 "你们好快!")

**对比**:
| 策略 | 承诺 | 实际交付 | 客户感受 |
|------|------|---------|---------|
| **激进承诺** | Q1做10个功能 | 完成6个 | 😡 失望 "怎么少了4个?" |
| **保守承诺** | Q1做6个功能 | 完成8个 | 😃 惊喜 "居然多做了2个!" |

### 💡 洞察2: "路线图不是承诺书" - 灵活调整的艺术

**实战场景** (2025某AI客服公司):

> "我们Q1路线图承诺 '上线工单系统', 销售已经拿着路线图去签了5个客户合同 (写明Q1交付).
>
> 但Q1市场突变: 竞品推出 'AI+人工混合模式', 我们客户开始流失. 如果继续做工单系统, Q2会失去20%客户.
>
> 决策: 紧急调整路线图, 砍掉工单系统, 全力做 '混合模式'. 但如何和已签约客户沟通?
>
> 沟通策略:
> 1. 坦诚说明: '市场变化, 我们发现混合模式比工单系统更重要'
> 2. 提供补偿: '免费延长3个月服务 + 优先试用混合模式'
> 3. 重新承诺: '工单系统延后到Q3, 但会做得更好'
>
> 结果: 5个客户中4个接受, 1个退款 (但避免了20%客户流失的更大损失)."

**教训**:
1. **路线图是战略工具, 不是法律合同** - 市场变化时, 必须调整
2. **提前沟通 > 事后道歉** - 越早告知客户, 信任损失越小
3. **补偿机制** - 如果客户因路线图变更受影响, 提供补偿 (如延长服务/折扣)

**何时必须调整路线图?**
- ✅ 竞品重大功能上线, 威胁核心业务
- ✅ 客户集中反馈某功能, 流失风险高
- ✅ 技术突破 (如新模型API), 原计划功能可秒杀
- ⚠️ CEO个人喜好 → 需要评估, 不能盲目调整

### 💡 洞察3: "用数据说话" - 路线图优先级的科学决策

**传统做法**: PM拍脑袋决定优先级 → "我觉得A功能更重要"

**科学做法**: 用数据 + 框架决策

**实战案例** (某AI写作助手, 2026年Q1):

**待决策功能** (只能选2个做):
1. AI润色功能
2. 多语言翻译
3. 模板库
4. 协作编辑

**数据收集**:
```
功能调研数据 (1000个活跃用户)

功能需求度 (用户投票):
- AI润色: 650票 (65%)
- 多语言: 200票 (20%)
- 模板库: 480票 (48%)
- 协作: 150票 (15%)

使用频率预估 (类似产品数据):
- AI润色: 每天5次/人
- 多语言: 每周1次/人
- 模板库: 每周3次/人
- 协作: 每月2次/人

付费意愿 (问卷):
- AI润色: 40%愿意为此付费
- 多语言: 25%
- 模板库: 15%
- 协作: 30%

开发成本 (研发评估):
- AI润色: 4周
- 多语言: 6周
- 模板库: 3周
- 协作: 10周
```

**RICE分数计算**:
| 功能 | Reach (用户%) | Impact (1-3) | Confidence (%) | Effort (周) | RICE分数 | 排名 |
|------|--------------|-------------|---------------|-----------|----------|------|
| AI润色 | 65% | 3 (高频+高付费意愿) | 90% | 4 | 43.9 | 🥇1 |
| 模板库 | 48% | 2 (中频+低付费) | 80% | 3 | 25.6 | 🥈2 |
| 多语言 | 20% | 2 (低频+中付费) | 70% | 6 | 4.7 | 🥉3 |
| 协作 | 15% | 2 (低频+中付费) | 60% | 10 | 1.8 | 4 |

**决策**: Q1做AI润色 + 模板库 (RICE分数最高的2个)

**反思**: 如果没有数据, PM可能因为 "多语言很酷" 或 "协作是趋势" 而选错, 浪费资源.

## ✅ 自测题

1. **产品路线图和项目计划的区别是什么? 请从时间粒度、关注重点、灵活性3个维度对比.**
   <details>
   <summary>查看答案</summary>

   | 维度 | 产品路线图 (Roadmap) | 项目计划 (Project Plan) |
   |------|---------------------|----------------------|
   | **时间粒度** | 季度/月 (3-18个月) | 周/天 (1-3个月) |
   | **关注重点** | Why (为什么做) + What (做什么) | How (怎么做) + When (什么时候) |
   | **灵活性** | 高 (可调整优先级) | 低 (需按计划执行) |

   简单记忆: 路线图是 "战略地图" (方向), 项目计划是 "作战手册" (执行).
   </details>

2. **Now-Next-Later路线图为什么适合AI产品? 请列举2个原因.**
   <details>
   <summary>查看答案</summary>

   原因:
   1. **技术变化快**: AI模型3-6个月就有重大更新 (如GPT-4 → GPT-4.5), 固定时间线的路线图容易过时. Now-Next-Later不承诺具体时间, 可以随时调整.

   2. **用户需求不确定**: AI产品还在教育市场阶段, 用户需求快速变化 (如Kimi发现用户更需要长文本而非聊天). Now-Next-Later可以快速验证NEXT的功能, 根据数据决定是否移到NOW.

   额外原因: 竞争激烈, 需要快速响应竞品动作, Now-Next-Later保留灵活性.
   </details>

3. **"承诺保守, 交付超预期" 的黄金法则是什么? 为什么要这样做?**
   <details>
   <summary>查看答案</summary>

   黄金法则:
   - 对外承诺 = 70%确定 + 保守时间 (如内部预计2个月, 对外说3个月)
   - 内部规划 = 120%野心 + 激进时间 (挑战团队极限)
   - 差额 = 惊喜空间 (提前完成, 客户觉得超预期)

   为什么?
   - **管理客户期待**: 承诺少了完成不了, 客户失望; 承诺多了超额完成, 客户惊喜
   - **降低风险**: 留buffer应对技术难题/人员变动等不可控因素
   - **提升信任**: 持续超预期交付, 客户信任度提升

   案例: 承诺Q1做6个功能, 实际完成8个 → 客户惊喜; 承诺10个, 完成6个 → 客户失望.
   </details>

4. **如果CEO要求在路线图里加一个 "他觉得很酷" 的功能, 但数据显示用户需求度低, 你该如何应对?**
   <details>
   <summary>查看答案</summary>

   应对策略 (分步骤):

   1. **理解CEO意图**: 问 "这个功能解决什么问题? 对公司战略有什么帮助?"
      - 可能CEO看到了你没看到的趋势
      - 也可能CEO只是一时兴起

   2. **展示数据**: 用RICE模型/用户调研数据说明该功能优先级低
      - "我们调研了1000个用户, 只有5%需要这个功能"
      - "这个功能需要8周开发, 但RICE分数只有1.2, 远低于其他功能"

   3. **提出折中方案**:
      - 方案A: 做MVP验证 (2周快速版本, 看数据再决定是否深入)
      - 方案B: 放到LATER (未来考虑, 但不占用NOW/NEXT资源)
      - 方案C: 如果CEO坚持, 问 "可以砍掉哪个现有功能?" (让CEO做取舍)

   4. **最终决策**: 如果CEO坚持且愿意承担风险 → 执行, 但设定验证指标 (如2周后看留存率, 低于X%就下线)
   </details>

### 练习4: 路线图变更决策

**场景**: 你的Q1路线图原本是 "功能A + 功能B", 但Q1刚开始, 竞品突然上线了功能C, 用户开始流失.

**数据**:
- 功能A: 已开发50%, 还需3周
- 功能B: 还没开始, 需4周
- 功能C (竞品刚上线): 如果不做, 预计Q1流失15%用户; 开发需5周

**你只有7周时间 (Q1剩余时间)**, 如何调整路线图?

**选项**:
- A. 继续做A + B, 不管竞品 (保持原计划)
- B. 砍掉B, 做A + C (应对竞品)
- C. 砍掉A, 做B + C (激进调整)
- D. 只做C, 砍掉A和B (极端应对)

**任务**: 选择一个方案, 并说明:
1. 为什么选这个?
2. 如何向团队/客户/销售沟通这个变更?
3. 有什么风险? 如何缓解?
