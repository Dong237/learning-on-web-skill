## 1. 优先级排序的本质

### 1.1 为什么优先级排序如此重要?

**残酷现实**: 100个需求, 你只能做10个 (资源有限)

**PM的核心价值** = 选对这10个, 让产品成功

**失败案例** (2024某AI创业公司):
- 产品经理: 同时做了20个功能 (都想要)
- 结果: 20个功能都是半成品, 没有一个做到极致
- 用户: "功能很多, 但都不好用", 3个月后倒闭

**成功案例** (Kimi):
- 产品经理: All-in长文本阅读, 其他功能砍掉
- 结果: 长文本体验吊打竞品, 用户增长迅速
- 用户: "Kimi = 长文本专家", 形成清晰定位

### 1.2 优先级排序的3个层次

```
第1层: 战略优先级 (Strategic)
"今年公司战略是什么? 哪些方向对战略有帮助?"

第2层: 功能优先级 (Feature)
"战略方向下, 先做哪个功能?"

第3层: 任务优先级 (Task)
"这个功能内部, 先做哪个细节?"
```

**案例: 某AI办公助手**

```
战略优先级:
✅ P0: 企业市场 (今年目标签100家企业)
⏸️ P1: C端市场 (暂时不做, 资源有限)

功能优先级 (企业市场下):
✅ P0: 团队协作功能 (企业核心需求)
✅ P0: 权限管理 (企业必须)
⏸️ P1: 数据分析 (Nice to have)
⏸️ P2: 个性化主题 (锦上添花)

任务优先级 (团队协作功能下):
✅ P0: 多人同时编辑 (核心体验)
✅ P0: 评论功能 (沟通必需)
⏸️ P1: @提醒 (重要但非必须)
⏸️ P2: 表情包 (可以以后加)
```

## 2. 5大经典优先级框架

### 2.1 RICE模型 (最常用)

**由Intercom公司提出, 适合数据驱动的团队**

```
RICE分数 = Reach × Impact × Confidence / Effort

Reach (覆盖率): 多少用户会用到? (数量/百分比)
Impact (影响度): 对用户价值有多大? (1=小, 2=中, 3=大)
Confidence (信心度): 有多确定? (0-100%)
Effort (工作量): 需要多少人天/人周? (越小越好)
```

**实战案例: 某AI写作助手功能排序**

| 功能 | Reach | Impact | Confidence | Effort | RICE | 优先级 |
|------|-------|--------|-----------|--------|------|--------|
| AI润色 | 80% | 3 | 90% | 4周 | 54 | 🥇 P0 |
| 模板库 | 60% | 2 | 80% | 3周 | 32 | 🥈 P0 |
| 多语言 | 30% | 2 | 70% | 6周 | 7 | 🥉 P1 |
| 协作编辑 | 20% | 3 | 50% | 10周 | 3 | P2 |

**RICE的优点**:
- ✅ 量化决策, 减少主观性
- ✅ 全面考虑 (价值+成本+风险)
- ✅ 易于沟通 (可以给老板看数据)

**RICE的缺点**:
- ⚠️ 需要数据支持 (创业公司早期可能没数据)
- ⚠️ Impact是主观评分 (仍有拍脑袋成分)
- ⚠️ 忽略了战略意义 (有些功能虽然RICE低, 但战略必须)

---

### 2.2 KANO模型 (用户满意度视角)

**由日本学者狩野纪昭提出, 分析功能对用户满意度的影响**

**KANO 5类需求**:

```
┌─────────────────────────────────────────┐
│ 1️⃣ 基本型需求 (Must-be)               │
│ "没有用户不满, 有了用户觉得理所当然"   │
│ 案例: AI产品的"能对话"                 │
├─────────────────────────────────────────┤
│ 2️⃣ 期望型需求 (Performance)           │
│ "做得越好, 用户越满意 (线性关系)"     │
│ 案例: AI回复速度 (越快越好)           │
├─────────────────────────────────────────┤
│ 3️⃣ 兴奋型需求 (Attractive)            │
│ "没有用户不在意, 有了用户超级惊喜"    │
│ 案例: Kimi长文本 (推出时让人惊艳)     │
├─────────────────────────────────────────┤
│ 4️⃣ 无差异需求 (Indifferent)           │
│ "有没有都无所谓"                       │
│ 案例: AI产品的"暗黑模式" (很少人在意)│
├─────────────────────────────────────────┤
│ 5️⃣ 反向需求 (Reverse)                 │
│ "有了反而用户不满"                     │
│ 案例: AI强制登录才能用 (用户讨厌)    │
└─────────────────────────────────────────┘
```

**KANO优先级策略**:

```
优先级排序:
P0: 基本型需求 (必须做, 否则用户不用)
P0: 期望型需求中的高优先级 (如速度/准确度)
P1: 兴奋型需求 (差异化, 但需验证)
P2: 期望型需求中的低优先级 (锦上添花)
P3: 无差异需求 (不做)
❌: 反向需求 (坚决不做)
```

**2026中国AI产品KANO分析**:

| 功能 | KANO类型 | 说明 | 优先级 |
|------|---------|------|--------|
| 能对话 | 基本型 | 没有就不是AI产品 | P0 |
| 回复速度快 | 期望型 | 越快越好 (3秒 vs 10秒, 用户明显感知) | P0 |
| 长文本阅读 | 兴奋型 (2024) → 基本型 (2026) | Kimi带火后, 用户已习惯 | P0 |
| 多模态 (图+文) | 兴奋型 (2025) → 期望型 (2026) | 逐渐成为标配 | P1 |
| 个性化皮肤 | 无差异型 | 用户不在意AI长什么样 | P3 |
| 广告 (免费版) | 反向需求 | 用户讨厌广告, 宁可付费 | ❌ |

**KANO动态变化**: 兴奋型需求会随时间变成期望型 → 最终变成基本型
- 2024: 长文本是兴奋型 (Kimi独有)
- 2025: 长文本是期望型 (通义/文心也有了)
- 2026: 长文本是基本型 (没有会被吐槽)

### 2.3 MoSCoW模型 (敏捷团队最爱)

**源自敏捷开发, 快速分类需求**

```
M - Must have (必须有)
   如果没有, 产品无法发布

S - Should have (应该有)
   重要但非必须, 可以下个版本做

C - Could have (可以有)
   Nice to have, 资源充足再做

W - Won't have (不会有)
   明确不做, 避免浪费讨论时间
```

**案例: AI客服产品MVP规划**

```
Must have (P0 - 必须):
✅ 智能问答 (核心功能)
✅ 人工客服转接 (AI不会的交给人)
✅ 微信渠道接入 (80%中国用户用微信)

Should have (P1 - 应该):
📋 数据分析报表 (帮客户优化)
📋 多渠道接入 (网页/电话)

Could have (P2 - 可以):
💡 语音对话 (多模态)
💡 情绪识别 (高级功能)

Won't have (不做):
❌ 视频客服 (需求太少)
❌ 区块链存证 (过度设计)
```

**MoSCoW的优点**:
- ✅ 简单直接, 5分钟可完成分类
- ✅ 团队容易达成共识
- ✅ 明确 "不做" 列表 (避免无意义争论)

**MoSCoW的缺点**:
- ⚠️ 过于粗糙 (M/S/C内部如何排序?)
- ⚠️ 容易被滥用 ("都是Must have!")

### 2.4 Value vs Effort 矩阵 (2×2象限图)

**最直观的可视化优先级工具**

```
        高价值
          ↑
    Q2    │    Q1
  复杂但  │  快速上线
  价值高  │  高价值低成本
──────────┼──────────→ 低成本
    Q3    │    Q4
  低优先级│  填充型任务
  价值低  │  成本低但价值也低
          ↓
        低价值
```

**4象限策略**:

| 象限 | 价值 | 成本 | 策略 | 优先级 |
|------|------|------|------|--------|
| **Q1** | 高 | 低 | 🎯 立刻做 (Quick Wins) | P0 |
| **Q2** | 高 | 高 | 📅 规划做 (Long-term) | P1 |
| **Q3** | 低 | 高 | ❌ 不做 (Avoid) | ❌ |
| **Q4** | 低 | 低 | 🛠️ 填充 (Fill-ins) | P2 |

**案例: AI产品功能分布图**

```
         高价值
           ↑
    ┌──────┼──────┐
    │ 私有化│ AI   │
    │ 部署  │ 摘要 │ ← Q1 (快速上线)
    │(Q2) │(Q1)  │
    ├──────┼──────┤
    │ 暗黑  │ Bug  │
    │ 模式  │ 修复 │ ← Q4 (填充)
    │(Q3) │(Q4)  │
    └──────┼──────┘
           ↓
         低价值

AI摘要 (Q1): 高价值 + 2周工作量 → 立刻做
私有化部署 (Q2): 高价值 + 3个月工作量 → Q3再做
Bug修复 (Q4): 低价值 (非致命) + 1天 → 有空就做
暗黑模式 (Q3): 低价值 + 2周 → 不做
```

**Value-Effort的优点**:
- ✅ 视觉化, 一目了然
- ✅ 适合团队讨论 (大家一起贴便签纸)
- ✅ 快速识别 "性价比最高" 的功能

**Value-Effort的缺点**:
- ⚠️ 价值和成本都是主观判断
- ⚠️ 忽略了风险 (Confidence)

### 2.5 ICE模型 (简化版RICE)

**适合快速决策, 不需要复杂计算**

```
ICE分数 = Impact × Confidence × Ease

Impact (影响力): 对目标的影响 (1-10分)
Confidence (信心度): 确定性 (1-10分)
Ease (容易度): 实施难易 (1-10分, 越容易分越高)
```

**案例: 快速评估3个需求**

| 需求 | Impact | Confidence | Ease | ICE分数 | 排序 |
|------|--------|-----------|------|---------|------|
| AI自动回复 | 9 | 8 | 7 | 504 | 🥇 1 |
| 数据看板 | 6 | 9 | 8 | 432 | 🥈 2 |
| 多语言支持 | 7 | 5 | 4 | 140 | 🥉 3 |

**ICE vs RICE**:
- RICE更精确 (需要具体数据)
- ICE更快速 (5分钟可评估10个需求)

## 3. 平衡多方利益相关者

### 3.1 优先级冲突的常见场景

**场景1: 销售 vs 产品**

| 角色 | 诉求 | 逻辑 |
|------|------|------|
| **销售** | "客户要XX功能, 不做就不签约" | 短期收入 |
| **产品** | "XX功能和战略不符, 不做" | 长期价值 |

**解决方案**:
```
决策框架:
1️⃣ 评估客户价值:
   - 这个客户ARR多少? (如果是大客户, 可以定制)
   - 这个需求有多少客户要? (如果只有1个客户, 不做)

2️⃣ 评估战略契合度:
   - 是否偏离产品定位? (如AI写作工具, 客户要加视频会议, 偏离)
   - 是否有复用性? (只能给这个客户用 vs 所有客户都能用)

3️⃣ 决策:
   - 大客户 + 战略契合 + 可复用 → 做
   - 小客户 + 战略不符 + 定制化 → 不做, 或收定制费
```

**2026案例** (某AI SaaS公司):
- 销售: "某金融客户要私有化部署, 100万合同"
- 产品: "私有化需要3个月, 团队只有10人"
- 决策: 做! 因为:
  1. 100万ARR = 全年目标的20%
  2. 金融行业有10+潜在客户, 可复用
  3. 私有化是企业市场趋势, 战略契合

---

**场景2: 老板 vs PM**

| 角色 | 诉求 | 逻辑 |
|------|------|------|
| **老板** | "我觉得AI语音功能很酷, 加上!" | 个人喜好 |
| **PM** | "用户调研显示只有5%需要语音" | 数据驱动 |

**解决方案**:
```
沟通话术:
"老板, 我理解语音功能很有前景. 我建议:
1️⃣ 先做MVP验证 (2周快速版), 看用户数据
2️⃣ 如果日活用户>30%使用, 我们Q2深入做
3️⃣ 如果<10%使用, 我们暂时不投入更多资源

这样既验证了想法, 又不影响当前核心功能开发."
```

**关键**: 不要直接说 "No", 而是用 "MVP验证" 给老板台阶下.

---

**场景3: 技术债 vs 新功能**

| 角色 | 诉求 | 逻辑 |
|------|------|------|
| **研发** | "代码太乱了, 需要重构, 否则以后更慢" | 长期效率 |
| **运营** | "用户在流失, 赶紧做新功能拉回来" | 短期增长 |

**解决方案**:
```
技术债务评估矩阵:

紧急度 (是否影响业务):
- P0: 线上故障 / 严重性能问题 → 立刻修
- P1: 开发效率降低50%+ → 必须重构
- P2: 代码不优雅但能用 → 慢慢优化

策略:
- 每个sprint预留20%时间修技术债 (如2周sprint, 2天修债务)
- 重构和新功能并行 (如 "本周做新功能A, 顺便重构相关模块")
```

## 4. 技术债务的优先级管理

### 4.1 什么是技术债务?

**技术债务 (Technical Debt)** = 为了快速上线, 欠下的代码质量债

**案例**:
- **借债**: 为了赶Q1上线, 代码写得很乱, 没有单元测试
- **还债**: Q2花2周重构代码, 补测试
- **利息**: 如果不还债, Q3每次改需求都要花2倍时间 (效率降低)

### 4.2 技术债的4个层次

| 层次 | 严重度 | 影响 | 优先级 | 案例 |
|------|--------|------|--------|------|
| **P0级** | 致命 | 影响用户 / 阻塞开发 | 立刻修 | 数据库随时崩溃 / 内存泄漏 |
| **P1级** | 严重 | 开发效率降低50%+ | 本季度修 | 代码耦合严重, 改一处牵连十处 |
| **P2级** | 中等 | 开发效率降低20% | 慢慢修 | 缺少单元测试, 改代码胆战心惊 |
| **P3级** | 轻微 | 影响小 | 有空再修 | 代码命名不规范, 但能看懂 |

### 4.3 技术债务的20%法则

**实战策略** (被多家硅谷公司验证):
```
每个sprint / 每个季度:
- 80%时间: 做新功能 (业务增长)
- 20%时间: 还技术债 (长期健康)

具体执行:
- 2周sprint → 2天修技术债
- 1季度 (3个月) → 2周重构
```

**2026中国AI公司案例** (某独角兽):
- Q1-Q2: 疯狂做功能, 代码债务堆积
- Q3: 发现开发速度降低60%, 每个需求从1周变3周
- Q4: 停止新功能, 全员重构1个月
- 结果: Q4收入为0 (因为没新功能), 投资人不满

**教训**: 别欠太多债, 否则一次性还债太痛苦.

## 5. 2026中国AI产品优先级实战

### 5.1 AI产品的优先级特殊挑战

**挑战1: 需求爆炸**
- 传统SaaS: 每月10个需求
- AI产品: 每天10个想法 (AI能做的太多了!)

**应对**: 更严格的优先级筛选, 只做RICE分数>50的

---

**挑战2: 技术不确定性**
- 传统开发: 需求确定 → 2周交付 (确定性高)
- AI开发: "试试看能不能做" → 可能1周, 可能1个月

**应对**: 用 "Time-boxing" (时间盒)
```
Time-boxing策略:
- 给每个探索性需求设定时间上限 (如2周)
- 2周后强制决策: 继续 or 放弃
- 避免无限期研究
```

---

**挑战3: 竞品压力**
- 竞品3天上线新功能 (调用API很快)
- 你是否要跟进? (打乱原计划)

**应对**: 建立 "快速响应池"
```
资源分配:
- 70%: 计划内功能 (路线图)
- 20%: 技术债务
- 10%: 快速响应池 (应对竞品/紧急需求)
```

---

### 5.2 2026中国AI产品优先级实战案例

**背景**: 某AI办公助手, Q1需要从20个需求中选5个做

**候选需求**:

| ID | 需求 | Reach | Impact | Confidence | Effort | RICE | KANO类型 | 利益相关者 |
|----|------|-------|--------|-----------|--------|------|---------|-----------|
| 1 | AI会议纪要 | 70% | 3 | 90% | 4周 | 47.3 | 期望型 | 核心用户强烈要求 |
| 2 | 飞书集成 | 50% | 3 | 80% | 6周 | 20 | 基本型 | 销售承诺3个大客户 |
| 3 | AI邮件回复 | 60% | 2 | 70% | 3周 | 28 | 期望型 | 运营建议 |
| 4 | 数据看板 | 30% | 2 | 90% | 5周 | 10.8 | 无差异 | 老板想要 |
| 5 | 多语言支持 | 20% | 2 | 60% | 8周 | 3 | 兴奋型 | 1个海外客户 |
| 6 | 语音输入 | 40% | 3 | 50% | 10周 | 6 | 兴奋型 | 技术团队想试 |
| 7 | 性能优化 | 100% | 2 | 100% | 2周 | 100 | 基本型 | 技术债 (加载慢) |
| 8 | Bug修复 | 100% | 1 | 100% | 1周 | 100 | 基本型 | 技术债 (崩溃率高) |
| 9 | 模板库 | 50% | 2 | 80% | 3周 | 26.7 | 期望型 | 用户调研 |
| 10 | API开放 | 10% | 3 | 60% | 12周 | 1.5 | 兴奋型 | 未来生态 |

**决策过程**:

```
第1轮: RICE排序 (数据驱动)
排名: 7 (性能) = 8 (Bug) > 1 (会议纪要) > 3 (邮件) > 9 (模板) > ...

第2轮: KANO调整 (用户满意度)
- 7和8是基本型 (必须做)
- 1、3、9是期望型 (做了用户满意度提升)
- 2虽然RICE低, 但是基本型 (企业客户必须)

第3轮: 利益相关者权衡
- 2 (飞书集成): 销售承诺3个大客户 (ARR共150万), 必须做
- 4 (数据看板): 老板要求, 但RICE低+KANO无差异 → 说服老板Q2再做

第4轮: 资源匹配 (只有8周时间)
- 7 (性能): 2周 ✅
- 8 (Bug): 1周 ✅
- 2 (飞书): 6周 (有点紧) ✅
- 1 (会议纪要): 4周 (但和飞书冲突, 选一个)

最终决策 (Q1做5个):
✅ 7. 性能优化 (2周) - P0技术债
✅ 8. Bug修复 (1周) - P0技术债
✅ 2. 飞书集成 (6周, 2个人) - P0战略必须
✅ 9. 模板库 (3周) - P1高性价比
✅ 3. AI邮件回复 (3周, 和模板库并行) - P1高RICE

延后到Q2:
⏸️ 1. AI会议纪要 (RICE高, 但Q1时间不够)
⏸️ 4. 数据看板 (说服老板Q2做)
❌ 5/6/10: RICE太低, 暂不考虑
```

**沟通话术** (向老板解释为什么数据看板延后):

> "老板, 我理解数据看板对决策很重要. 但我们Q1有3个P0任务 (性能/Bug/飞书集成), 是签约大客户的前提. 数据看板的RICE分数是10.8, 远低于会议纪要 (47.3). 我建议Q2优先做会议纪要 (70%用户需要), 然后Q3做数据看板. 您看可以吗?"

## 实战洞察

### 💡 洞察1: "永远的P0陷阱" - 如何打破

**常见场景**:
- 销售: "这个需求是P0, 客户要签100万合同"
- 运营: "这个需求是P0, 用户在流失"
- 老板: "这个需求是P0, 竞品有了我们没有"
- 技术: "这个Bug是P0, 不修服务器会崩"

**结果**: 10个P0需求, 团队崩溃 (什么都是P0 = 什么都不是P0)

**解决方案: P0重新定义**

```
真正的P0 (只有3种):
1️⃣ 线上故障 / 安全漏洞 (影响所有用户)
2️⃣ 大客户签约前提 (ARR>50万 + 合同已签 + 明确deadline)
3️⃣ 战略必须 (CEO明确要求 + 对公司生死攸关)

其他都是P1/P2:
- 用户流失 → P1 (严重但不致命)
- 竞品有了 → P1 (重要但可计划)
- 小Bug → P2 (慢慢修)
```

**实战技巧**: 建立 "P0审批机制"
```
谁能定义P0?
- 只有CEO / CTO / CPO 可以定义P0
- 其他人提P0需求, 必须邮件抄送CEO审批

P0配额制:
- 每个季度最多5个P0
- 新P0加入, 必须有旧P0降级
```

### 💡 洞察2: "数据不是万能的" - 何时忽略数据

**场景1: 用户不知道自己要什么**

**案例** (乔布斯名言):
> "如果我问用户要什么, 他们会说 '更快的马', 而不是汽车."

**2026 AI案例** (Kimi长文本):
- 用户调研: "你需要AI读长文档吗?" → 80%用户说 "不需要, 我自己能读"
- 实际数据: Kimi上线后, 长文本使用率60%+, 成为核心功能

**教训**: 创新功能需要 "vision驱动", 而非数据驱动. 数据只能优化已知需求, 无法发现未知需求.

---

**场景2: 数据会骗人**

**案例** (某AI产品):
- 数据: "暗黑模式使用率30%, 需求很高!"
- PM决定: 投入2周开发暗黑模式
- 结果: 上线后使用率仍30%, 留存率/付费率无提升
- 反思: 30%用户开启暗黑模式, 但这不是核心需求, 不影响付费决策

**教训**: 看数据要看 "北极星指标" (如留存率/付费率), 而非虚荣指标 (如使用率).

---

**场景3: 战略 > 数据**

**案例** (某AI公司2025):
- 数据: C端用户100万, B端用户1000家, C端明显更大
- 战略: All-in B端 (企业服务毛利高, 可持续)
- 决策: 砍掉C端90%功能, 资源全投B端
- 1年后: B端ARR从500万增长到5000万, 公司盈利

**教训**: 数据告诉你 "现在是什么", 战略告诉你 "未来去哪". 优先级要服从战略.

---

### 💡 洞察3: "说No的艺术" - 5种拒绝话术

**场景1: 拒绝销售的客户需求**
- ❌ 直接拒绝: "这个需求不做, 没有意义"
- ✅ 数据说话: "我们调研了100个客户, 只有2个要这个功能, RICE分数1.5, 暂不排期"
- ✅ 提供替代: "这个功能暂时不做, 但可以用XX功能部分替代"

---

**场景2: 拒绝老板的想法**
- ❌ 直接拒绝: "老板, 这个想法不靠谱"
- ✅ MVP验证: "老板, 这个想法很有前景, 我们先做2周MVP验证, 如果数据好再深入"
- ✅ 延后承诺: "这个功能很重要, 但Q1资源已满, 我们Q2优先做, 您看可以吗?"

---

**场景3: 拒绝技术团队的重构**
- ❌ 直接拒绝: "现在没时间重构, 先做功能"
- ✅ 部分满足: "全量重构3个月太长, 我们先花2周重构最痛的模块, 其他慢慢来"
- ✅ 建立机制: "以后每个sprint预留20%时间修技术债, 不要堆积"

---

**场景4: 拒绝用户的需求**
- ❌ 直接拒绝: "这个需求我们不做"
- ✅ 解释原因: "这个功能和我们产品定位不符, 我们聚焦XX场景, 您可以试试YY产品"
- ✅ 收集反馈: "感谢建议, 我们记录下来, 如果有更多用户提, 我们会考虑"

---

**场景5: 拒绝竞品压力**
- ❌ 盲目跟进: "竞品有了, 我们也做!"
- ✅ 战略定位: "竞品做XX, 但我们差异化是YY, 不需要跟进"
- ✅ 数据验证: "我们先看用户有没有因为这个功能流失, 如果流失率<5%, 暂不做"

## ✅ 自测题

1. **RICE模型的4个维度是什么? 请用一个AI产品功能举例计算RICE分数.**
   <details>
   <summary>查看答案</summary>

   RICE = Reach × Impact × Confidence / Effort

   案例: AI简历优化的 "面试问题预测" 功能
   - Reach (覆盖率): 60%用户会用 (求职者都需要)
   - Impact (影响度): 3分 (高价值, 帮用户拿offer)
   - Confidence (信心度): 70% (技术可行但不100%确定)
   - Effort (工作量): 4周

   RICE = 60% × 3 × 70% / 4 = 0.315 (× 100 = 31.5分)

   对比其他功能 (如简历模板 RICE=50), 面试预测优先级较高.
   </details>

2. **KANO模型的5类需求是什么? 请举例说明 "兴奋型需求" 如何随时间变成 "基本型需求".**
   <details>
   <summary>查看答案</summary>

   KANO 5类: 基本型 / 期望型 / 兴奋型 / 无差异型 / 反向型

   案例: AI长文本阅读能力的演变
   - 2024年初 (Kimi推出): **兴奋型需求**
     - 用户: "哇, 居然能读100万字! 太惊喜了!"
     - 竞品没有, Kimi独家
   - 2025年中: **期望型需求**
     - 通义/文心也有了, 用户开始对比 "谁的长文本更好"
     - 有 → 满意, 没有 → 不满
   - 2026年初: **基本型需求**
     - 所有AI助手都有长文本, 用户默认 "应该有"
     - 有 → 理所当然, 没有 → 用户吐槽 "怎么连长文本都不支持?"

   教训: 兴奋型需求的窗口期很短 (6-12个月), 要快速抓住.
   </details>

3. **什么是 "永远的P0陷阱"? 如何打破?**
   <details>
   <summary>查看答案</summary>

   永远的P0陷阱 = 所有需求都被标记为P0, 团队无法聚焦

   原因:
   - 销售: "客户要的都是P0"
   - 老板: "我的想法都是P0"
   - 技术: "所有Bug都是P0"

   打破方法:
   1. **重新定义P0**: 只有3种才是真P0
      - 线上故障/安全漏洞
      - 大客户签约前提 (ARR>50万 + 合同已签)
      - 战略生死攸关 (CEO明确)
   2. **P0审批机制**: 只有CEO/CTO/CPO可定义P0
   3. **P0配额制**: 每季度最多5个P0, 新增必须有旧的降级

   案例: 某公司Q1有15个P0 → 重新评估后只有3个真P0, 其他降为P1/P2, 团队效率提升50%.
   </details>

4. **技术债务的20%法则是什么? 为什么要预留时间还技术债?**
   <details>
   <summary>查看答案</summary>

   20%法则:
   - 每个sprint/季度, 80%时间做新功能, 20%时间还技术债
   - 具体: 2周sprint → 2天修债务; 1季度 → 2周重构

   为什么要还技术债?
   1. **效率降低**: 债务堆积 → 改需求时间变长 (1周 → 3周)
   2. **质量下降**: 没测试 → 经常出Bug → 用户流失
   3. **团队士气**: 工程师讨厌在烂代码上工作 → 离职

   反例 (某AI公司):
   - Q1-Q2: 疯狂做功能, 0%时间修债务
   - Q3: 发现开发速度降60%, 被迫停止新功能
   - Q4: 全员重构1个月, 收入为0

   教训: 别欠太多债, 小步快还 (20%法则) 比一次性还债更健康.
   </details>
