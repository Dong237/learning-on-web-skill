## TL;DR - 5秒速览

- **PM的核心价值** 不是自己做事, 而是 "让合适的人在合适的时间做合适的事"
- **跨职能协作** (Cross-functional Collaboration) 是PM的必修课: 研发/设计/运营/销售/客服
- **2026中国协作工具**: 飞书/钉钉主导, PingCode/Teambition/Worktile分场景使用

- **常见陷阱**: "传声筒PM" (只会传达需求, 不懂各职能专业) vs "独行侠PM" (什么都自己干)

## 1. PM的协作本质

### 1.1 PM是 "无权的领导者"

**残酷现实**:
- PM没有直接下属 (研发/设计/运营都不汇报给你)
- PM不能命令别人 "你必须做XX"
- PM靠的是 **影响力** (Influence) 而非权力 (Authority)

**PM的角色定位**:

```
              PM (产品经理)
                  ↓
     影响力驱动 (Influence, not Authority)
                  ↓
    ┌─────────┬────────┬─────────┬─────────┐
    ↓         ↓        ↓         ↓         ↓
  研发团队   设计团队  运营团队  销售团队  客服团队
    ↓         ↓        ↓         ↓         ↓
  (写代码)  (画原型)  (拉新)   (卖产品)  (答疑)
```

**如何建立影响力?**
1. **专业能力**: 懂业务 + 懂用户 + 懂数据, 让团队信服你的判断
2. **同理心**: 理解各职能的难处 (如研发的技术债, 设计的美学追求)
3. **沟通能力**: 清晰表达需求, 不模棱两可
4. **结果导向**: 做出来的产品真的有价值, 团队愿意跟你干

---

### 1.2 跨职能协作的3大挑战

**挑战1: 目标不一致**

| 职能 | 核心KPI | 关注点 | 与PM的冲突 |
|------|---------|--------|----------|
| **研发** | 代码质量 / 按时交付 | 技术债务 / 架构优雅 | PM要功能, 研发要重构 |
| **设计** | 用户体验 / 设计美感 | 细节打磨 / 品牌一致性 | PM要快速上线, 设计要完美 |
| **运营** | 用户增长 / 活跃留存 | 拉新 / 促活 / 转化 | PM聚焦产品, 运营要运营工具 |
| **销售** | 签单金额 / 客户数量 | 客户需求 / 竞品对比 | PM做长期, 销售要短期签单功能 |

**解决方案**: 建立共同的北极星指标 (North Star Metric)
- 不是 "PM的KPI vs 研发的KPI"
- 而是 "产品的核心指标" (如DAU / 付费转化率 / NPS)
- 所有职能围绕这个指标协作

---

**挑战2: 语言不通**

**案例**: PM说 "我要一个AI功能", 研发听不懂具体要什么

| PM的语言 | 研发的理解 | 正确的沟通方式 |
|---------|----------|--------------|
| "AI智能推荐" | "用什么算法? CF还是深度学习?" | "根据用户历史行为, 推荐相关内容, 准确率>70%, 响应<500ms" |
| "界面要简洁" | "具体多简洁? 去掉什么?" | "首页只保留3个核心功能入口, 其他折叠到'更多'" |
| "性能要好" | "多好? 什么指标?" | "首屏加载时间<2秒, P95响应时间<1秒" |

**教训**: PM要学会 "翻译" - 把用户语言翻译成技术语言

---

**挑战3: 优先级冲突**

**场景**: PM说 "A功能P0", 研发说 "技术债也是P0"

**解决方案**: 透明化优先级决策过程
```
决策会议 (每周1次):
- 参会: PM + 研发负责人 + 设计负责人
- 议程: Review本周待做需求 + RICE排序
- 输出: 一个明确的优先级列表 (P0/P1/P2)
- 公示: 贴到飞书/钉钉群, 所有人可见
```

## 2. 与研发团队的协作

### 2.1 PM与研发的5大协作场景

**场景1: 需求评审 (Requirement Review)**

**目的**: 让研发理解需求的 "Why" (为什么做) 和 "What" (做什么), 而非直接告诉 "How" (怎么做)

**错误做法** (指挥式PM):
```
PM: "你们要用React做前端, 用Node.js做后端, 数据库用MySQL..."
研发: "你懂技术吗? 别指手画脚!"
```

**正确做法** (协作式PM):
```
PM: "我们需要一个AI对话功能, 用户输入问题, 1秒内返回答案.
     核心体验是'快', 技术方案你们看着选, 我信任你们的专业."
研发: "OK, 我们评估后, 用Streaming SSE实现实时返回, 比WebSocket更简单."
```

**需求评审checklist**:
- ✅ 背景: 为什么要做这个功能? (业务目标)
- ✅ 用户故事: 谁在什么场景下要做什么? (User Story)
- ✅ 验收标准: 什么情况下算做完? (Acceptance Criteria)
- ✅ 非功能需求: 性能/安全/兼容性要求
- ⚠️ 不要: 具体技术方案 (交给研发决定)

---

**场景2: 技术方案评审 (Technical Design Review)**

**目的**: PM理解技术方案的权衡, 提前预判风险

**PM需要问的问题**:
1. "这个方案有什么风险? 如果失败Plan B是什么?"
2. "开发时间是多久? 有没有更快的实现方式?"
3. "这个方案会产生技术债吗? 未来扩展性如何?"
4. "依赖外部服务吗? (如GPT API) 如果断供怎么办?"

**案例** (2025某AI创业公司):
- 研发方案: 100%依赖OpenAI GPT-4 API
- PM追问: "如果OpenAI断供 (中美关系), 我们怎么办?"
- 研发调整: 主用GPT-4, 备用通义千问API, 降低风险

---

**场景3: 每日站会 (Daily Standup)**

**目的**: 同步进度, 发现阻碍

**格式** (每人1-2分钟):
1. 昨天做了什么?
2. 今天计划做什么?
3. 有什么阻碍? (PM帮忙解决)

**PM的角色**: **扫除障碍者 (Blocker Remover)**
- 研发: "我需要设计稿, 但设计师还没给"
- PM: "我去催设计师, 今天下午给你"

**2026中国公司实践**:
- 字节/阿里: 早上10点站会 (线下或飞书视频)
- 腾讯: 异步站会 (飞书文档每天更新进度, 不开会)

---

**场景4: Sprint演示 (Sprint Demo)**

**目的**: 展示本周成果, 收集反馈

**参会**: PM + 研发 + 设计 + 运营 + (可选)客户

**流程**:
1. 研发演示功能 (10分钟)
2. 大家提反馈 (5分钟)
3. PM确认验收标准是否达成 (5分钟)

**验收标准** (Acceptance Criteria):
```
功能: AI摘要
验收标准:
✅ 用户上传PDF, 10秒内返回摘要 (性能)
✅ 摘要准确率>80% (质量, 通过人工抽检10篇文档)
✅ 支持中英文 (功能)
❌ 不支持图片/表格解析 (明确不做)
```

---

**场景5: 回顾会议 (Sprint Retrospective)**

**目的**: 反思本周哪里做得好, 哪里要改进

**3个问题**:
1. What went well? (哪些做得好? 下次继续)
2. What didn't go well? (哪些不好? 要改进)
3. Action items? (具体改进措施)

**案例** (某AI团队回顾):
- 做得好: 需求评审清晰, 研发没有返工
- 不好: 设计稿延迟2天, 影响开发进度
- Action: 下个sprint, 设计稿提前3天交付, PM监督

### 2.2 PM如何赢得研发的尊重?

**研发最讨厌的PM行为**:
1. ❌ 朝令夕改 (今天说做A, 明天改做B)
2. ❌ 不懂技术还指导技术 ("你们为什么不用XX框架?")
3. ❌ 不尊重工作量 ("这个功能很简单, 1天做完吧?")
4. ❌ 甩锅 (出问题就说 "研发没做好")
5. ❌ 需求不清晰 (研发做完发现和预期不一样, 返工)

**研发最喜欢的PM行为**:
1. ✅ 需求清晰 (User Story + 验收标准明确)
2. ✅ 尊重专业 (技术方案让研发决定)
3. ✅ 承担责任 (产品失败, PM不甩锅给研发)
4. ✅ 扫除障碍 (帮研发协调设计/运营资源)
5. ✅ 懂一点技术 (不需要写代码, 但知道什么能做什么不能做)

**实战建议**: PM应该学习的技术知识
- ✅ 基础: 前端/后端/数据库是什么, 各有什么作用
- ✅ API: RESTful API / GraphQL 是什么
- ✅ AI基础: 大模型的能力边界 (什么能做, 什么不能做)
- ✅ 性能指标: QPS / P95延迟 / 并发 是什么意思
- ⚠️ 不需要: 会写代码 (PM的核心能力不是coding)

## 3. 与设计团队的协作

### 3.1 PM与设计的协作流程

```
PM                      设计师                    输出
────────────────────────────────────────────────────
1. 需求文档 (PRD)   →   理解需求
                    ←   澄清问题              澄清会议

2. 用户研究         →   设计方向探索
   (User Research)  ←   竞品分析              设计brief

3. 信息架构         →   低保真原型
   (IA)             ←   交互方案              Wireframe

4. 评审交互         →   高保真设计
   (Review)         ←   视觉设计              Mockup

5. 开发跟进         →   设计走查
   (Dev handoff)    ←   验收还原度            上线
```

### 3.2 PM与设计的3大冲突及解决

**冲突1: 完美主义 vs 快速迭代**

| 设计师 | PM |
|--------|-----|
| "这个按钮圆角4px太丑, 要改成6px" | "用户根本不在意圆角, 赶紧上线" |
| "品牌一致性很重要, 不能妥协" | "MVP先上线, 以后再优化" |

**解决方案**: 分级设计 (Tiered Design)
```
P0 (核心体验, 必须完美):
- 首页 / 核心功能交互 / 品牌主色调
→ 设计师主导, PM不干涉

P1 (重要但可迭代):
- 二级页面 / 边缘功能
→ 先快速上线, 后续优化

P2 (无关紧要):
- 404页面 / 系统设置页
→ 用默认设计, 不浪费时间
```

---

**冲突2: 用户体验 vs 商业目标**

| 设计师 | PM |
|--------|-----|
| "广告太多, 破坏用户体验" | "不放广告怎么赚钱?" |
| "弹窗太频繁, 用户烦" | "不弹窗转化率低" |

**解决方案**: A/B测试 + 数据说话
```
假设: 减少弹窗会降低转化率
实验: A组 (1次弹窗) vs B组 (3次弹窗)
数据:
- A组: 转化率2%, 用户留存率70%
- B组: 转化率3%, 用户留存率50%

结论: B组短期转化高, 但长期留存低 (用户被烦跑了)
决策: 用A组方案, 长期价值更高
```

---

**冲突3: 创新设计 vs 用户习惯**

| 设计师 | PM |
|--------|-----|
| "我设计了一个全新的导航方式, 很酷!" | "用户习惯了传统导航, 学习成本高" |

**解决方案**: 创新遵循 "80/20法则"
```
80%: 遵循行业标准 (如底部Tab导航, 用户不用学习)
20%: 创新设计 (差异化, 但不影响核心体验)

案例 (某AI产品):
- 80%标准: 聊天界面 (和微信一样, 用户熟悉)
- 20%创新: AI自动分类对话 (新功能, 但不影响基本聊天)
```

### 3.3 PM如何给设计师写Brief?

**设计Brief模板**:

```markdown
# 设计Brief: AI摘要功能

## 1. 背景与目标
- 用户痛点: 长文档阅读耗时, 用户需要快速抓住重点
- 产品目标: 让用户10秒内了解文档核心内容
- 成功指标: 摘要使用率>60%, 满意度>4分/5分

## 2. 目标用户
- 主要用户: 25-35岁知识工作者 (产品经理/分析师/研究员)
- 使用场景: 上班路上用手机快速浏览报告, 或开会前快速了解文档

## 3. 核心功能
- 用户上传PDF/Word文档 (≤50MB)
- AI生成3-5句话的摘要 (中英文均支持)
- 用户可调整摘要长度 (短/中/长 3档)
- 支持导出摘要 (复制/分享)

## 4. 设计约束
- 技术限制: 生成时间5-10秒 (需要loading状态)
- 品牌调性: 专业 + 简洁 (ToB产品, 不要太花哨)
- 平台: 优先Web端, 后续做移动端

## 5. 参考竞品
- Kimi长文本阅读 (交互流畅, 学习)
- Notion AI摘要 (简洁, 但不够智能)

## 6. 时间要求
- 低保真原型: 3天 (周三)
- 高保真设计: 1周 (下周一)
- 开发时间: 2周 (设计稿交付后)
```

## 4. 与运营/销售/客服的协作

### 4.1 PM与运营的协作

**运营的诉求**: 拉新 / 促活 / 转化 / 留存

**常见冲突**:
- 运营: "我要一个积分系统, 刺激用户活跃"
- PM: "积分系统开发3个月, 优先级低"

**解决方案**: 区分 "产品功能" vs "运营工具"
```
产品功能 (PM负责):
- 核心价值功能 (如AI对话/摘要/翻译)
- 所有用户都会用

运营工具 (运营负责, PM支持):
- 活动/优惠券/积分/推送
- 临时性, 快速上下线

策略:
- PM提供 "运营后台" (如优惠券配置系统)
- 运营自己配置活动, 不需要每次找研发开发
```

**2026中国AI公司案例** (豆包):
- PM做核心AI能力
- 运营用 "火山引擎增长工具" 自己配置活动 (如邀请好友得会员)

---

### 4.2 PM与销售的协作

**销售的诉求**: 快速签单, 客户要什么就做什么

**常见冲突**:
- 销售: "客户要私有化部署, 这单100万, 必须做!"
- PM: "私有化偏离战略, 团队只有10人, 做不过来"

**解决方案**: 建立 "客户需求评估机制"
```
客户需求评估表:

1️⃣ 客户价值:
   - ARR (年度合同金额): ¥___万
   - 行业代表性: 是/否 (如金融行业标杆客户)
   - 复用性: 多少其他客户也需要? ___家

2️⃣ 战略契合度:
   - 是否符合产品定位? 是/否
   - 是否有技术积累价值? 是/否

3️⃣ 资源评估:
   - 开发时间: ___周
   - 需要几个人? ___人
   - 影响其他需求吗? 是/否

决策矩阵:
- ARR>50万 + 战略契合 + 可复用 → 做 (P0)
- ARR<10万 + 战略不符 + 定制化 → 不做, 或收定制费
```

**实战案例** (某AI SaaS):
- 客户A: 要定制AI模型, ARR 5万 → 拒绝 (太小)
- 客户B: 要私有化部署, ARR 100万, 金融行业标杆 → 做 (P0)

---

### 4.3 PM与客服的协作

**客服的价值**: 最了解用户真实痛点 (用户不会跟PM说, 但会跟客服抱怨)

**协作方式**:
```
1️⃣ 每周客服反馈会 (30分钟)
   - 客服: 汇报本周用户Top 3问题
   - PM: 判断是产品问题还是用户教育问题
   - 输出: 本周修复的Bug列表 + 下周优化计划

2️⃣ 客服工单系统对接
   - 客服在飞书/钉钉提工单, 标记 "产品问题"
   - PM每天review, 高频问题纳入优化计划

3️⃣ PM定期听客服电话 (每月1次)
   - 理解用户真实困惑
   - 发现产品体验问题
```

**2026案例** (某AI客服产品):
- 客服反馈: 30%用户问 "AI回复不准怎么办?"
- PM调查: 发现是用户不会写提示词
- 解决方案: 增加 "提示词模板" 功能, 问题降到5%

## 5. 2026中国AI公司协作工具生态

### 5.1 主流协作工具对比

| 工具 | 定位 | 优势 | 劣势 | 适用公司 |
|------|------|------|------|---------|
| **飞书** | 字节系, 一体化办公 | ✅ IM+文档+日历+会议一体 ✅ AI能力强 | ⚠️ 非字节系公司接受度一般 | 字节系/AI创业公司 |
| **钉钉** | 阿里系, 中小企业主导 | ✅ 覆盖率高 (中小企业) ✅ 审批流程强 | ⚠️ 产品体验一般 | 传统企业/中小企业 |
| **企业微信** | 腾讯系, 私域流量 | ✅ 微信生态无缝对接 | ⚠️ 协作功能弱于飞书/钉钉 | ToC业务的公司 (如电商) |
| **Lark** | 飞书国际版 | ✅ 海外团队适用 | ⚠️ 国内用户少 | 出海公司 |

**项目管理工具**:

## 6. 远程协作与异步沟通

### 6.1 远程协作的3个层次

**Level 1: 完全线下** (传统模式)
- 所有人在办公室
- 问题面对面沟通
- 适合: 小团队 (<10人), 创业早期

**Level 2: 混合办公** (2026主流)
- 核心团队在办公室, 部分远程
- 重要会议线下, 日常远程
- 适合: 中型团队 (10-50人), 成熟公司

**Level 3: 完全远程** (Remote-first)
- 所有人远程, 无固定办公室
- 100%异步沟通
- 适合: 开源项目, 全球化团队

---

### 6.2 异步沟通 (Async Communication) 原则

**什么是异步沟通?**
- 同步沟通: 实时会议/电话 (双方同时在线)
- 异步沟通: 文档/邮件/录屏 (不需要同时在线)

**异步沟通的优势**:
1. ✅ 跨时区协作 (如北京团队 + 硅谷团队)
2. ✅ 减少会议 (不用为了5分钟问题开30分钟会)
3. ✅ 信息留存 (文档永久保存, 会议内容容易忘)
4. ✅ 深度工作 (不被频繁打断)

**异步沟通的工具**:

| 场景 | 同步工具 | 异步替代 | 案例 |
|------|---------|---------|------|
| 需求评审 | 30分钟会议 | 飞书文档 + 评论 | PM写需求文档, 研发异步评论, 24小时内回复 |
| 设计反馈 | 面对面讨论 | Figma评论 | PM在设计稿上标注问题, 设计师改完更新 |
| 技术方案 | 技术评审会 | 飞书文档 + Loom录屏 | 研发录屏讲解方案, PM看完评论 |
| 进度同步 | 每日站会 | 飞书日报 | 每人每天更新进度到文档, 不开会 |

---

### 6.3 异步协作的最佳实践

**实践1: 文档优先 (Documentation-first)**

```
传统模式 (会议驱动):
1. PM开会讲需求 (30分钟)
2. 研发听完忘了, 再问一遍
3. 来回沟通3-4轮才清楚

异步模式 (文档驱动):
1. PM写清晰的需求文档 (1小时)
2. 研发异步阅读, 在文档评论提问 (10分钟)
3. PM 24小时内回复 (5分钟)
结果: 节省80%时间, 信息留存
```

**文档模板** (飞书/Notion):
- 需求文档: PRD模板 (背景/目标/用户故事/验收标准)
- 技术方案: Tech Spec模板 (方案/权衡/风险)
- 会议纪要: 决策/Action Items/负责人

---

**实践2: 录屏替代会议 (Loom / 飞书妙记)**

**场景**: 设计师做了新的原型, 需要给PM讲解

**传统**: 约30分钟会议, PM和设计师同时在线

**异步**: 设计师录5分钟屏 (Loom), 边操作边讲解, PM有空时看, 在视频下评论反馈

**工具**:
- Loom: 录屏 + 评论 (国外主流)
- 飞书妙记: 录屏 + AI自动生成字幕和总结 (国内)

---

**实践3: 设定响应时间 (Response Time SLA)**

**问题**: 异步沟通如果没有时间约定, 会变成 "无限期等待"

**解决**: 建立响应时间规范
```
P0紧急: 2小时内响应 (如线上故障)
P1重要: 24小时内响应 (如需求澄清)
P2一般: 48小时内响应 (如功能建议)

工作时间: 9:00-21:00 (中国时区)
非工作时间: 不强制响应 (除非P0)
```

## 实战洞察

### 💡 洞察1: "PM不是传声筒" - 增值而非转发

**传声筒PM** (最失败的类型):
```
销售: "客户要XX功能"
PM: (转发给研发) "研发, 做XX功能"
研发: "为什么要做? 优先级多高?"
PM: "我也不知道, 销售说的"
研发: 💢 (心想: 这PM有啥用?)
```

**增值型PM** (优秀PM):
```
销售: "客户要XX功能"
PM: (调研) "这是哪个客户? ARR多少? 这个功能解决什么问题?"
    (分析) "类似需求还有3个客户提过, 总ARR 200万, RICE分数40"
    (决策) "这个需求Q2做, 优先级P0"
    (沟通研发) "我们Q2做XX功能, 原因是... 预期收益是... 技术方案你们看着来"
研发: ✅ (清楚为什么做, 愿意投入)
```

**PM的增值体现在**:
1. **需求分析**: 客户要的 ≠ 客户需要的 (深挖背后的真实需求)
2. **优先级决策**: 不是客户说什么就做什么 (用RICE/KANO评估)
3. **方案设计**: 给研发 "What" 和 "Why", 不是 "How"
4. **风险预判**: 提前发现问题 (如技术风险/资源冲突)

---

### 💡 洞察2: "同理心是PM的超能力"

**场景**: 研发延期, PM的两种反应

**反应A (缺乏同理心)**:
```
PM: "为什么延期? 你们承诺的1周交付!"
研发: "中间遇到技术难题, 需要重构..."
PM: "重构是你们的问题, 不关我事, 必须按时交付!"
研发: 💢 (以后不愿意跟你合作)
```

**反应B (同理心)**:
```
PM: "延期了吗? 遇到什么困难?"
研发: "XX模块的架构有问题, 不重构以后更麻烦..."
PM: "我理解重构的重要性. 这样, 我们延后3天, 但你们保证质量. 我去跟老板/客户解释延期原因."
研发: ✅ (感受到支持, 愿意加班搞定)
```

**同理心的3个维度**:
1. **理解难处**: 研发的技术债, 设计的美学追求, 运营的增长压力
2. **换位思考**: "如果我是研发, 我会怎么想?"
3. **共同承担**: PM不甩锅, 一起扛

**实战案例** (某AI创业公司PM):
> "我们Q1延期了, 老板很生气. 我没有甩锅给研发, 而是说 '是我需求评审不够细致, 导致返工'. 研发很感动, 主动加班2周补进度. 后来那个研发跟我说, 他愿意跟我做任何项目."

---

### 💡 洞察3: "会议是必要的恶" - 如何开高效会议

**残酷现实**: 大部分会议是浪费时间
- 30%的会议: 根本不需要开 (发个文档就行)
- 50%的会议: 效率低下 (30分钟能解决的开了2小时)
- 只有20%的会议: 真正有价值

**如何减少无效会议?**

**原则1: No Agenda, No Meeting**
```
必须有议程的会议:
✅ 需求评审 (议程: 讲解需求 + Q&A)
✅ Sprint回顾 (议程: 好/坏/改进)
❌ "我们聊聊产品方向" (太模糊, 改成文档)
```

---

**原则2: 能异步就异步**

| 场景 | 传统 (会议) | 异步替代 | 节省时间 |
|------|-----------|---------|---------|
| 需求宣讲 | 30分钟会议 | 飞书文档 + Loom录屏 | 节省60% (参会人自己看) |
| 进度同步 | 15分钟站会 | 飞书日报 | 节省100% (不开会) |
| 设计反馈 | 1小时评审会 | Figma评论 | 节省80% (设计师改完PM审) |

---

**原则3: 会议必须有产出**

**无产出会议** (浪费时间):
```
会议内容: 大家讨论了1小时产品方向
结果: 没有结论, "下次再聊"
```

**有产出会议** (高效):
```
会议产出 (会后5分钟整理):
✅ 决策: Q2做A功能, 不做B功能
✅ Action Items:
   - PM写需求文档 (周三前)
   - 设计出原型 (下周一)
   - 研发评估工作量 (下周三)
✅ 负责人: 每个Action都有明确负责人
✅ 时间: 每个Action都有deadline
```

**会议纪要模板** (飞书文档):
```markdown
# 会议纪要: Q2产品规划

时间: 2026-01-15 14:00-15:00
参会: PM (张三) / 研发负责人 (李四) / 设计负责人 (王五)

## 决策
1. Q2做A功能 (AI摘要), 不做B功能 (多语言翻译)
2. 设计风格: 简洁专业, 参考Kimi

## Action Items
| 任务 | 负责人 | Deadline | 状态 |
|------|--------|---------|------|
| 写需求文档 | PM张三 | 1月18日 | ⏳进行中 |
| 设计原型 | 设计王五 | 1月22日 | 📋待做 |
| 评估工作量 | 研发李四 | 1月25日 | 📋待做 |

## 下次会议
时间: 1月26日 14:00
议程: 评审需求文档 + 设计原型
```

### 练习3: 跨职能冲突调解

**任务**:

### 练习4: 异步协作方案设计

- 每天开3个会 (需求评审/进度同步/设计反馈), 浪费2小时
