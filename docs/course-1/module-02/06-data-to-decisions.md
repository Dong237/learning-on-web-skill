## 数据驱动≠数据独裁

### 三种常见误区

**误区1:迷信数据,忽视直觉**

```
反例:某AI产品团队的失败案例(2024)

PM提议:"增加AI语音对话功能"
数据团队:"现有用户调研,需求度不高,不做"

2个月后:
竞品推出语音功能,抢走大量用户
事后分析:用户不提=不需要?错!是用户想象不到这种体验

教训:
创新功能无法完全用历史数据预测,需要产品直觉+小范围MVP验证
```

**误区2:拍脑袋决策,无视数据**

```
反例:某AI陪伴产品的灾难(2025)

创始人(技术极客):"我觉得用户需要AI的哲学思考能力"
PM:"数据显示大部分用户只聊日常,要不要验证一下?"
创始人:"不用,我相信这个方向,全力开发!"

3个月后上线:
- 使用率极低
- 用户反馈:"太抽象了,我就想聊聊天放松"
- 浪费大量研发成本

教训:
个人偏好≠用户需求,重大决策必须数据验证
```

**误区3:数据瘫痪,不敢决策**

```
反例:某AI写作产品的拖延症(2026)

团队纠结:"到底要不要增加多人协作功能?"

分析了3个月:
- 做了5轮用户调研
- 看了10份竞品报告
- 跑了20个AB测试

结果:
竞品已经上线并占领市场,自己还在"再等等数据"

教训:
数据永远不会100%确定,决策需要在"80%信息"时下注
```

### 正确理解数据驱动

**数据驱动决策的本质**:

```
          数据驱动决策
               ↓
    ┌──────────┴──────────┐
    │                     │
  定量数据              定性洞察
    │                     │
  (What/How Much)      (Why/How)
    │                     │
  埋点/AB测试          用户访谈
  仪表盘/漏斗          竞品分析
    │                     │
    └──────────┬──────────┘
               ↓
          产品决策
            (判断+行动)
```

**金句**:
> "数据告诉你'是什么',直觉告诉你'可能是什么',PM的价值是综合判断后敢于下注。"

## 优先级排序框架:RICE模型

### RICE模型详解

**公式**:
```
RICE分数 = (Reach × Impact × Confidence) / Effort

Reach(触达):   多少用户会受影响?
Impact(影响):  对用户价值有多大?
Confidence(信心): 有多确定这个预测?
Effort(成本):  需要多少人月?
```

**评分标准**:

| 维度 | 评分方法 | 示例 |
|------|---------|------|
| **Reach** | 季度内影响的用户数 | "功能优化影响100万DAU" = 1,000,000 |
| **Impact** | 3(巨大) / 2(高) / 1(中) / 0.5(小) / 0.25(微小) | "提升核心转化率"=3,"UI微调"=0.5 |
| **Confidence** | 100%(确定) / 80%(高信心) / 50%(中等) / 20%(低) | "有AB测试数据"=100%,"拍脑袋"=20% |
| **Effort** | 人月(Person-Months) | "前后端2人开发1个月"=2 |

### 实战案例:AI对话产品的需求优先级

**背景**:某AI对话产品收到5个需求,资源有限,如何排序?

| 需求 | Reach | Impact | Confidence | Effort | RICE分数 | 排序 |
|------|-------|--------|-----------|--------|---------|------|
| **A. 语音输入功能** | 500万(50%用户) | 2(高) | 80% | 3人月 | **2.67** | 🥇 1 |
| **B. AI联网搜索** | 800万(80%用户) | 3(巨大) | 50% | 6人月 | **2.00** | 🥈 2 |
| **C. 多轮对话优化** | 1000万(100%用户) | 1(中) | 100% | 2人月 | **5.00** | 🥇 1 |
| **D. UI皮肤切换** | 200万(20%用户) | 0.5(小) | 100% | 1人月 | **1.00** | 4 |
| **E. 支持PDF上传** | 300万(30%用户) | 2(高) | 60% | 4人月 | **0.90** | 5 |

**计算过程(需求C)**:
```
Reach = 1000万
Impact = 1 (中等影响,现有功能优化)
Confidence = 100% (已有AB测试验证,次留有显著提升)
Effort = 2人月

RICE = (1000万 × 1 × 100%) / 2 = 500万 / 2 = 5.00
```

**最终决策**:
1. **优先做C**(RICE=5.00):全量用户+高确定性+低成本 = 最高ROI
2. **其次做A**(RICE=2.67):高需求+合理成本
3. **再做B**(RICE=2.00):高价值但成本高,排第三
4. D、E暂缓,等下季度资源充足再考虑

<details>
<summary>深入了解:RICE模型的变体 - ICE、WSJF</summary>

**ICE模型**(简化版RICE)
```
ICE = Impact × Confidence / Effort
(忽略Reach,适合早期产品,用户基数小)
```

**WSJF模型**(敏捷开发常用)
```
WSJF = (Business Value + Time Criticality + Risk Reduction) / Effort

Business Value:商业价值
Time Criticality:时间敏感性(如监管要求、竞品压力)
Risk Reduction:降低风险程度

示例:"算法备案"需求
- Business Value: 5(不做会被下架)
- Time Criticality: 10(监管deadline)
- Risk Reduction: 8(避免合规风险)
- Effort: 2人月

WSJF = (5+10+8) / 2 = 11.5 (极高优先级)
```

**选择指南**:
- 创业公司早期:用ICE(简单直接)
- 中等规模产品:用RICE(全面均衡)
- 大厂敏捷团队:用WSJF(考虑时间风险)
</details>

### RICE模型的局限性

**无法量化的因素**:
- **战略意义**:某功能当前Impact低,但是未来战略方向
- **团队士气**:某需求能激励团队,提升凝聚力
- **技术债**:重构代码无直接用户价值,但长期必须做

**应对方法**:
```
RICE分数 + PM判断 = 最终决策

示例:
"重构AI推理引擎"RICE=0.5(用户无感知)
但PM判断:不重构,3个月后系统扛不住流量,全线崩溃
→ 决策:优先级提升到Top 3
```

## 数据讲故事的艺术

### 为什么需要"讲故事"?

**场景**:PM向老板汇报需求

```
❌ 数据罗列版(老板听完一脸懵)

PM:"我们的7日留存是18%,DAU/MAU是0.25,
    CSAT是76%,NPS是32,MAU环比增长5%..."

老板:"所以呢?你想要什么资源?"

PM:"呃...我觉得我们需要优化用户体验..."

老板:"具体怎么优化?为什么?"

PM:"..." (支支吾吾)
```

```
✅ 数据讲故事版(老板听完立即批资源)

PM:"老板,我们遇到一个紧急问题:

【问题】7日留存从25%暴跌到18%,流失了30%新用户

【影响】按当前增长,3个月后DAU会下降15%,Q2目标无法完成

【根因】深度访谈20个流失用户,发现70%在首次使用就遇到bug
        (展示用户原话录音:"AI根本听不懂我说什么,卸载了")

【方案】投入2人月优化新手引导,预期次留提升到40%(AB测试验证)

【回报】挽回30万流失用户,增加¥60万订阅收入(ARPU ¥2)

【要资源】1个前端+1个AI工程师,2周完成"

老板:"批准,立即开始!"
```

### 数据讲故事的5步法

**Step 1:设定背景 (Context)**
- "我们的产品处于什么阶段?目标是什么?"

**Step 2:抛出问题 (Problem)**
- "遇到了什么挑战?数据怎么显示的?"

**Step 3:挖掘根因 (Root Cause)**
- "为什么会这样?(用数据+访谈+竞品分析)"

**Step 4:提出方案 (Solution)**
- "我们要怎么做?(具体行动+资源需求)"

**Step 5:预测结果 (Expected Outcome)**
- "做了之后会怎样?(量化目标+验证方法)"

### 实战案例:AI对话产品的"响应速度"优化

**PM的汇报PPT** (数据故事范本)

```markdown
【第1页:问题】
"用户投诉激增:对话响应慢"
- 数据:用户满意度(CSAT)显著下降
- 客服工单:近期投诉量暴增,关键词"卡""慢""等半天"

【第2页:影响】
"如果不解决,将导致用户流失加剧"
- 趋势分析:响应速度与用户留存高度相关
- 商业损失:基于当前DAU和ARPU预估,潜在损失巨大

【第3页:根因】
"技术排查+用户访谈,找到3大原因"
1. 模型推理慢(首token响应时间过长) - 主要问题
2. 网络抖动(移动网络不稳定) - 次要问题
3. 服务器过载(高峰期负载过高) - 辅助问题

【第4页:解决方案】
分优先级逐一攻克
- P0(立即):扩容服务器,解决过载问题
- P1(本周):优化模型推理,大幅降低首token时间
- P2(下周):弱网优化,增加断点续传

【第5页:验证计划】
灰度小比例用户,监控核心指标
- 目标:CSAT回升至健康水平
- 监控:响应速度、崩溃率、留存率
- Timeline:本周灰度,数据达标后全量

【第6页:资源需求】
- 2个后端工程师(优化推理)
- 1个前端(弱网优化)
- 云服务器扩容预算

【第7页:总结】
"小投入解决大问题,ROI显著"
```

**为什么这个汇报成功?**
1. 数据清晰明确(CSAT下降,工单暴增等核心指标)
2. 商业影响量化(基于DAU和ARPU测算潜在损失)
3. 根因分析充分(技术+用户双重验证)
4. 方案可执行(分优先级,Timeline明确)
5. ROI清晰(小投入解决大问题)

## 决策框架:何时信数据,何时信直觉

### 决策矩阵

```
              数据可得性
                │
        低      │      高
                │
    ┌───────────┼───────────┐
    │           │           │
高  │  案例1    │   案例2   │
    │  依赖直觉  │  数据驱动  │
不  │  (创新功能)│ (功能优化) │
确  ├───────────┼───────────┤
定  │           │           │
性  │  案例3    │   案例4   │
    │  快速试错  │  深度调研  │
低  │  (MVP)   │ (重大决策) │
    │           │           │
    └───────────┴───────────┘
```

**案例1:高不确定性 + 数据少 → 产品直觉+MVP**
```
场景:开发全新AI功能"情绪陪伴"
现状:市场没有先例,用户调研也说不清需求
决策:PM凭直觉判断有价值,做最小MVP测试
方法:1周开发原型,灰度100用户,观察留存
```

**案例2:低不确定性 + 数据多 → 纯数据驱动**
```
场景:优化现有功能的按钮颜色
现状:用户基数大,可快速AB测试
决策:100%相信数据,哪个转化率高用哪个
方法:AB测试7天,样本10万,p<0.01即采纳
```

**案例3:低不确定性 + 数据少 → 快速试错**
```
场景:竞品推出新功能,跟不跟?
现状:功能成熟度未知,来不及调研
决策:快速抄袭MVP,2周上线,边做边看数据
方法:最小实现,灰度5%,不行就下线
```

**案例4:高不确定性 + 数据多 → 深度分析后决策**
```
场景:是否进入全新赛道(如AI搜索)
现状:市场大但竞争激烈,战略级决策
决策:3个月深度调研:市场规模、竞品、技术可行性
方法:二手研究+一手调研+技术预研,CEO最终拍板
```

### 何时敢于"逆数据"决策?

**乔布斯名言**:
> "用户不知道自己想要什么,直到你把它放在他们面前。"

**3种可以"逆数据"的场景**:

**场景1:创造新需求**
```
案例:iPhone问世前(2007)
调研:"你想要触屏手机吗?"
用户:"不,我要实体键盘,触屏不好用"

乔布斯:做!
结果:重新定义手机

启示:颠覆式创新无法用历史数据预测,需要愿景+执行力
```

**场景2:战略卡位**
```
案例:字节跳动做豆包(2023)
数据:AI对话市场已有ChatGPT、文心一言、通义千问,红海竞争
董事会:"数据显示很难突围,要不要做?"

字节:做!
理由:即使短期亏损,AI是未来战略高地,必须卡位

结果:豆包于[2023年8月推出](https://zh.wikipedia.org/zh-hans/%E8%B1%86%E5%8C%85_(%E8%81%8A%E5%A4%A9%E6%9C%BA%E5%99%A8%E4%BA%BA)),到[2025年12月DAU已破1亿](https://finance.sina.com.cn/roll/2025-12-24/doc-inhcwxmi9810374.shtml),成为国内领先AI应用
```

**场景3:团队信念**
```
案例:某AI教育产品的坚持(2025)
数据:前3个月留存惨淡,投资人要求砍项目
团队:"我们相信AI能改变教育,再给3个月"

行动:深度访谈100个用户,找到真实痛点,重新设计
结果:第5个月留存开始攀升,第8个月拿到B轮

启示:数据差不一定是方向错,可能是执行问题
```

**警告**:以上仅适用于"战略级决策",日常产品优化请务必数据驱动!

## 2026中国公司决策文化

### 字节跳动:AB测试信仰

**文化特点**:"大力出奇迹"变成"数据出奇迹"

**决策机制**:
1. **假设驱动**:PM提出假设,不是直接要资源
2. **小范围验证**:灰度1%用户,快速验证
3. **数据说话**:核心指标提升>3%且p<0.01,才全量
4. **快速迭代**:1周一个AB测试,3个月100+实验

**字节跳动的AB测试文化**
```
字节跳动以数据驱动著称,通过大量AB测试优化产品

AB测试实践:
- 每年进行数万次实验([抖音年均6-8万次](https://www.cnblogs.com/bytedata/p/16885511.html))
- 从功能设计到UI细节,都通过数据验证
- 典型流程:小范围灰度 → AB测试 → 数据分析 → 决策上线

核心原则:
- 不是"老板说了算",是"数据说了算"
- 假设驱动:PM提出假设,用实验验证
- 快速迭代:1周一个测试,持续优化

教训:AB测试适合成熟产品优化,用户基数大可快速验证
```

**适合场景**:
- ✅ 成熟产品优化(用户基数大,可快速AB)
- ✅ 短期决策(功能迭代、UI调整)
- ❌ 战略级决策(AB测试无法预测3年后趋势)

### 阿里巴巴:数据中台+政委文化

**文化特点**:"业务+数据+组织"三位一体

**数据中台架构**:
```
          业务层
    (淘宝/钉钉/通义千问)
            │
            ↓
        数据中台
    (统一数据/统一指标/统一工具)
            │
            ↓
      基础设施
    (MaxCompute/Hologres)
```

**决策机制**:
1. **数据民主化**:每个员工都能查数据(权限内)
2. **指标体系化**:北极星指标 → 一级指标 → 二级指标,层层拆解
3. **政委介入**:重大决策有"政委"(HR)参与,平衡数据与人性

**实战案例:钉钉的"已读功能"平衡术**
```
争论:要不要显示"消息已读"?

数据派:"已读能提升沟通效率"
人性派:"员工反馈,已读像监控,压力大"([2023年调查显示近八成网友支持取消](https://m.mp.oeeee.com/a/BAAFRD000020230714819123.html))
政委:"已读功能会影响组织文化,建议可选"

最终决策:
- 支持灵活配置(群聊可设置强制已读回执)
- 平衡效率与人性化
- 让团队根据文化自主选择

启示:B端产品需考虑组织文化,不能只看效率数据
```

**适合场景**:
- ✅ 复杂业务(多业务线,需统一数据)
- ✅ B端产品(涉及组织管理,需考虑人性)
- ❌ 快速创新(流程重,决策慢)

### 腾讯:产品经理文化+赛马机制

**文化特点**:"小团队试错,内部赛马"

**决策机制**:
1. **PM自主权大**:PM提案→leader批准→自己组队开干
2. **内部竞争**:同一需求,2个团队同时做,跑赢的全量上线
3. **用户体验优先**:数据重要,但"用户爽不爽"更重要

**实战案例:微信的"朋友圈三天可见"**
```
初期数据:使用比例较低
产品经理:"虽然用的人不多,但对隐私敏感用户很重要"

决策逻辑:
- 不是看多少人用(数据)
- 而是看对谁重要(用户价值)
- 保留功能,甚至优化交互

结果:
使用率持续增长,[2021年已有2亿用户设置三天可见](https://news.cctv.com/2023/11/23/ARTIxqqmrmsIbyhjF4R6ZAyZ231123.shtml),[调查显示63.9%用户使用该功能](https://news.cctv.com/2023/11/23/ARTIxqqmrmsIbyhjF4R6ZAyZ231123.shtml),成为微信最受欢迎的隐私设置
```

```markdown
3. 每个负面反馈给出可能的产品解决方案

```

**通义千问输出**:
```markdown
【分类统计】
- 正面:45% (90条) - 关键词:好用、快速、效果好
- 负面:40% (80条) - 关键词:生成失败、不像、太慢
- 中性:15% (30条) - 关键词:还行、可以、一般

【TOP 5负面反馈】
1. 生成失败(32次) - "输入Prompt后,提示生成失败"
   方案:优化失败重试机制,增加error提示
   优先级:高(影响核心体验)

2. 生成速度慢(28次) - "等了1分钟还没出图"
   方案:扩容GPU服务器,或降低默认分辨率
   优先级:高(用户流失点)

3. 效果不符预期(15次) - "生成的图和我描述的不一样"
   方案:提供Prompt优化建议,或增加参考图上传
   优先级:中(教育用户)

4. 无法修改细节(8次) - "想改某个局部,只能重新生成"
   方案:开发局部重绘(Inpainting)功能
   优先级:中(需求明确但开发成本高)

5. 版权担忧(5次) - "AI生成的图能商用吗?"
   方案:明确版权说明,提供商用授权选项
   优先级:低(少数用户,但长期重要)
```

**PM的下一步**:
1. 立即修复"生成失败"和"速度慢"(P0)
2. 用AB测试验证Prompt优化功能
3. 将"局部重绘"加入下季度Roadmap
4. 法务团队准备版权说明文档

### 场景2:用豆包做RICE优先级计算

**任务**:对比3个需求的RICE分数

**Prompt**:
```markdown
我有3个需求要评估优先级,请用RICE模型计算分数并排序:

需求A:语音输入功能
- Reach:500万用户(50%DAU)
- Impact:2(高,提升输入效率)
- Confidence:80%(有用户调研支持)
- Effort:3人月

需求B:多轮对话优化
- Reach:1000万用户(100%DAU)
- Impact:1(中,改善现有体验)
- Confidence:100%(已有AB测试数据)
- Effort:2人月

需求C:AI绘画功能
- Reach:300万用户(30%DAU,新功能估算)
- Impact:3(巨大,全新价值)
- Confidence:50%(市场不确定)
- Effort:6人月

请计算RICE分数,并给出推荐顺序和理由。
```

**豆包输出**:
```markdown
【RICE分数计算】

需求A:语音输入
RICE = (500万 × 2 × 80%) / 3 = 800万 / 3 = 2.67

需求B:多轮对话优化
RICE = (1000万 × 1 × 100%) / 2 = 1000万 / 2 = 5.00 ⭐

需求C:AI绘画
RICE = (300万 × 3 × 50%) / 6 = 450万 / 6 = 0.75

【推荐顺序】
1. 需求B:多轮对话优化(RICE=5.00)
2. 需求A:语音输入(RICE=2.67)
3. 需求C:AI绘画(RICE=0.75)

【决策理由】
- 需求B:全量用户受益,确定性高,成本低,ROI最高
- 需求A:差异化功能,中等成本,值得做
- 需求C:虽然Impact大,但不确定性高+成本高,建议先做MVP验证

【风险提示】
需求C的Confidence仅50%,建议:
1. 先投入1人月做原型
2. 灰度100用户测试
3. 如留存提升>10%,再全力开发
```

## 💡 实战洞察 (AI PM Practitioner Insights)

> **洞察1:警惕"数据迷信" - 数据只是工具,不是神谕**
>
> 2025年某AI客服团队被数据"绑架":
> - AB测试显示"蓝色按钮"比"绿色按钮"转化率高2%
> - 团队每天跑10个AB测试,优化按钮、文案、布局...
> - 3个月后:转化率提升15%,但用户满意度暴跌(NPS-10)
>
> **真相**:过度优化转化率,忽略用户长期价值
>
> **教训**:数据是"后视镜",只能看到过去;愿景是"望远镜",看到未来。两者都要。

---

> **洞察2:中国PM的"数据+关系"双重能力**
>
> 美国PM:数据驱动就够了,说服老板靠数据
> 中国PM:数据驱动+关系运营,说服老板靠数据+信任
>
> **实战经验**:
> - 数据再好,老板不信任你=白搭
> - 平时多沟通,建立信任,关键时刻老板才会支持你"逆数据"决策
> - 阿里的"政委文化"本质就是"数据+人性"平衡
>
> **启示**:技术能力是入场券,沟通能力是胜负手

## ✅ 自测题 (Self-Check Questions)

<details>
<summary>Q1: 用RICE模型评估以下需求,并排序优先级</summary>

**需求背景**:某AI写作产品,3个待开发需求

需求1:多人协作编辑
- Reach: 200万(20%用户,企业用户)
- Impact: 3(巨大,差异化功能)
- Confidence: 60%(市场调研,需求不确定)
- Effort: 8人月

需求2:Markdown导出
- Reach: 800万(80%用户)
- Impact: 1(中等,便利性功能)
- Confidence: 100%(用户强烈要求)
- Effort: 1人月

需求3:AI自动配图
- Reach: 500万(50%用户,内容创作者)
- Impact: 2(高,提升效率)
- Confidence: 80%(有竞品验证)
- Effort: 4人月

**答案**:

计算RICE分数:
- 需求1: (200万 × 3 × 60%) / 8 = 360万 / 8 = **0.45**
- 需求2: (800万 × 1 × 100%) / 1 = 800万 / 1 = **8.00** ⭐
- 需求3: (500万 × 2 × 80%) / 4 = 800万 / 4 = **2.00**

**优先级排序**:
1. 需求2(Markdown导出) - RICE=8.00,全量用户受益,成本极低
2. 需求3(AI配图) - RICE=2.00,高价值+中等成本
3. 需求1(多人协作) - RICE=0.45,虽然Impact大,但不确定性高+成本高,建议MVP验证

</details>

<details>
<summary>Q2: 场景题 - 数据与直觉冲突时,你如何决策?</summary>

**场景**:你负责AI对话产品,提出"增加语音输入"功能

- 用户调研:仅8%用户表示需要语音输入
- 竞品分析:主要竞品都没有语音功能
- 老板质疑:"数据这么差,为什么还要做?"

**你作为PM,如何说服老板?**

**参考答案**:

**Step 1:承认数据,但挑战数据**
"老板,8%的需求确实不高,但我认为这个数据有误导性:
- 用户从未体验过,无法评估价值(潜在需求)
- 调研时问'你需要吗',用户都说不需要;但给他用了,可能离不开"

**Step 2:提供支持证据**
"我做了3个补充调研:
1. 深度访谈10个用户,发现开车/做饭时想用AI但没法打字(场景需求)
2. 竞品虽没做,但语音AI(如小爱/Siri)MAU>1亿,市场已被教育
3. 我们的AI语音识别技术已成熟(技术就绪),成本可控"

**Step 3:降低风险**
"我建议用MVP验证:
- 投入1人月,做基础版本(仅语音输入,不做对话)
- 灰度5%用户,监控使用率和留存
- 如果使用率<1%,立即下线;如果>5%,全力开发"

**Step 4:量化收益**
"如果成功,预计:
- 吸引10%新用户(开车场景)
- 提升现有用户使用频次+20%(多场景使用)
- 差异化竞争优势(我们有,竞品没有)"

**Step 5:请求决策**
"综上,我认为值得用1人月试错。老板您看呢?"

**关键**:
- 不是"数据vs直觉",而是"当前数据vs潜在价值"
- 用MVP降低风险,让老板敢于下注
- 量化收益,给老板信心

</details>

<details>
<summary>Q3: 为什么字节跳动强调AB测试,而腾讯强调产品直觉?背后的业务逻辑是什么?</summary>

**答案**:

**字节跳动的业务特点**:
- 产品:抖音、今日头条(内容分发)
- 核心:算法推荐
- 用户基数:数亿DAU,样本充足
- 优化空间:无数细节可优化(推荐算法、UI、交互)

→ **适合AB测试**:
- 样本大,统计显著性容易达到
- 小改进累积=巨大收益(提升1%留存=千万用户)
- 创始人张一鸣是工程师思维,"相信数据>相信人"

**腾讯的业务特点**:
- 产品:微信、QQ(社交通讯)
- 核心:用户体验、情感连接
- 特点:功能成熟,大改动影响大(微信克制,轻易不改)
- 优化空间:更多在产品哲学,而非AB测试能解决

→ **适合产品直觉**:
- 社交产品难量化(情感、氛围无法AB测试)
- 张小龙强调"用完即走",与"提升使用时长"的数据导向相反
- 核心是"做对的事",而非"数据好看的事"

**启示**:
- 内容分发产品:数据驱动(抖音、淘宝、百度)
- 社交/工具产品:体验驱动(微信、Notion、Figma)
- AI产品:兼顾两者(技术靠数据,体验靠直觉)

</details>

<details>
<summary>Q4: 如何用AI(通义千问/豆包)辅助产品决策?它的局限性是什么?</summary>

**AI能做的(辅助)**:
1. 分析大量文本数据(用户反馈、行业报告)
2. 计算RICE/ICE分数
3. 生成决策框架(如"XX场景下应该考虑哪些因素?")
4. 头脑风暴(给出多种解决方案)

**AI做不到的(需要人类)**:
1. 战略判断(AI没有商业vision)
2. 风险承担(AI不会为决策负责)
3. 人性洞察(AI理解不了文化、情感)
4. 跨领域创新(AI基于已有数据,难以突破)

**实战建议**:
- ✅ 用AI处理"可量化"任务(数据分析、计算、总结)
- ✅ 用AI扩展思路(头脑风暴、竞品调研)
- ❌ 不要让AI做"最终决策"(它是助手,不是老板)
- ❌ 不要盲信AI输出(需人工验证,尤其数据来源)

**案例**:
某PM让通义千问"帮我决定要不要做语音功能"
AI输出:"建议做,因为用户调研显示..."

问题:AI基于输入数据(历史),无法预测未来(创新)
正确用法:让AI"列出做/不做的pros and cons",人类综合判断

</details>
